# 双仓库协作与决赛提交流程规范

> ⚠️ **部分内容已过期**：本文档 §3.1 中"直接 push main"的工作流是比赛期**单人开发**的简化流程。
> 引入团队协作后，**所有合并到 main 的变更必须通过 Pull Request**，详见 [分支与 Commit 规范](../../team/commit-conventions.md)。
> 本文档中关于"两个代码库的定位与同步"的描述仍然有效。

## 1. 背景与核心定位

由于初赛已经结束，官方比赛仓库已被冻结。进入维护期并备战决赛阶段，为了让决赛评审团可以透明、精准地评估我们在赛后维护期的开发轨迹与工作量，本项目采用专业软件工程中常见的**“双代码库协同”**架构。

这两个核心代码库的职能与定位划分如下：

### 1.1 Dev 开发仓库（“兵工厂”）
* **项目身份：** `StudySolo-Dev` (当前本地绑定的默认 `origin` 远程端)
* **核心角色：** 持续迭代的主阵地、敏捷开发实验室。
* **行为准则：** 
  * 它是我们项目生命周期日常流转的核心。不要觉得它是“替补”，它是真正的**幕后主力**。
  * 所有的日常开发、API 联调、报错崩溃、实验性修改都在这个空间消化。
  * 绝对包容碎片化的 Commit 历史（如 `fix minor typo`）以及试验性质的分支。随时 Push，不要有任何心理负担。

### 1.2 官方比赛仓库（“阅兵广场”）
* **项目身份：** 记录初赛冻结版本的旧仓库。
* **核心角色：** 面向评委的成果展厅、商业 Release 稳定发版点。
* **行为准则：**
  * 保持主分支（`main`）的比赛初赛快照，**严禁强推覆盖**。
  * 它充当了我们 Dev 主库稳定节点的“映像备份”。评委将在此仓库审核维护期的真实代码增量与工程规范。

## 2. 桥接双库：环境配置规则

直接在当前的本地项目库（`StudySolo-Dev` 的副本目录）下，将旧官方仓库作为全新的远程节点（`Remote`）挂载进来。无需复制两份本地文件夹。

```bash
# 挂载官方比赛旧仓库（将URL替换为实际旧仓库地址）
git remote add official https://github.com/AIMFllys/[比赛旧仓库名].git

# 检查挂载状态，必须同时具备 origin(Dev) 和 official(展示库) 两个远程端
git remote -v
```

## 3. 稳健提交流程（CI/CD 流）

### 3.1 🟢 日常研发流（日常模式）
* **场景：** 每日高频代码编写。
* **动作：** 无视比赛仓库，完全留在 Dev 生态闭环内。
  ```bash
  git add .
  git commit -m "feat/fix: 简要描述"
  git push origin main
  ```

### 3.2 🔵 里程碑同步流（评审汇演模式）
* **场景：** 功能实现大闭环、周末阶段性总结、或准决赛前需要让评审专家查阅进度。
* **目标：** 将大量的颗粒度 Commit 无损输送给旧仓库，供评委审查开发轨迹。

**Step 1: 推送隔离的验收分支**
将本地当前最新的 `main` 沿一条安全通道，推送到官方库里的一个独立分支：
```bash
git push official main:feature/finals-maintenance
```

**Step 2: 运用 Pull Request 发挥汇报价值**
* 强推或直接合并是不规范的。需前往官方比赛库的 GitHub 端，向 `main` 分支提出一条 Pull Request (PR)。
* 将 PR 视为**第一手文字答辩材料**。在 PR Description 中清晰罗列维护期内解决的核心业务问题与技术创新。
* **价值落地：** 评委只需点开此 PR，所有的代码增量、技术复杂度（`Files changed`）及开发时间线（`Commits`）即可成为决赛拿分的有力自证依据。
