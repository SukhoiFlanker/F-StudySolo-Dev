# 现有节点功能补全总则

> 创建时间：2026-03-27
> 编码要求：UTF-8
> 前置：必须先完成 [00-节点与插件分类判断.md](./00-节点与插件分类判断.md)

适用场景：仓库里已经存在节点，但该节点仍缺少真实功能、配置能力、节点内操作或可读预览，需要按真实架构继续补全。

---

## 1. 目标

现有节点补全不是“微调文案”，而是把节点做到真正可用。

最低完成标准：

- 有真实执行能力
- 有 `config_schema`
- 有节点内配置入口
- 有画布预览
- 有执行面板 `compact`
- 有错误双视图

---

## 2. 真实架构基线

- Prompt：`backend/app/nodes/_base.py`
- Node manifest：`/api/nodes/manifest`
- 节点配置存储：`node.data.config`
- 节点执行入参：`NodeInput.node_config`
- 执行面板：`ExecutionTraceDrawer`

任何补全都必须在这条链路上收口，禁止再造第二套节点配置协议。

---

## 3. 实施顺序

1. 先分类：A / B / C
2. 补 manifest：`config_schema / supports_upload / supports_preview / output_capabilities`
3. 补节点配置抽屉：可编辑 `node.data.config`
4. 补后端真实消费：执行时必须读取 `node_config`
5. 补画布预览与 `compact`
6. 补验收与文档

---

## 4. 简单节点与升级边界

本文件默认处理“简单节点补全”，即：

- 不新增独立插件页
- 不新增左侧导航入口
- 不新增复杂业务表

如果补全过程中出现以下任一情况，立即升级为 C 型：

- 需要独立页面
- 需要新数据库表
- 需要侧边栏入口
- 改动范围明显超出节点自身与既有 Service/API

---

## 5. 现有节点补全清单模板

每补一个节点，至少写清楚以下 6 项：

1. 节点分类：A1 / A2 / B-Tool / B-Search / B-Augmented / C
2. 真实执行能力：后端是否闭环
3. 配置项：`config_schema`
4. 节点内操作：是否上传 / 导出 / 预览 / 重试
5. 预览验收：画布 + `compact`
6. 错误验收：NodeResultSlip + ExecutionTraceDrawer

---

## 6. 编码门禁

- 所有文档、Prompt、Markdown 模板必须使用 UTF-8、无 BOM、LF
- 禁止中文转义
- 禁止提交乱码截图式内容
- 修改后必须抽样检查中文标题、表格、代码块和长段落

---

> 📌 **核心原则**：现有节点补全必须让“节点本身”成为功能主入口，而不是继续依赖旧页面或隐藏流程。
