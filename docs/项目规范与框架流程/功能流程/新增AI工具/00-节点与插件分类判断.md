# 新增 AI 工具：节点与插件分类判断

> 最后更新：2026-03-27
> 编码要求：UTF-8
>
> **必读**：在动任何代码之前，必须先用本文件确定你要新增的是哪种类型。错误的分类会导致走错 SOP，造成架构混乱。

---

## 0. 先判断是“新增”还是“补全”

从 2026-03-27 起，本目录的 SOP 同时覆盖两类工作：

- **新增节点**：仓库里还不存在该节点，需要新建后端节点包、前端类型与渲染器。
- **现有节点补全**：节点已存在，但需要补真实能力、补 `config_schema`、补节点内配置入口、补画布预览、补执行面板 `compact`。

**硬规则**：

1. 现有节点补全也必须先做 A / B / C 分类，不能因为“节点已存在”就跳过分类。
2. 若补全过程中出现独立页面、左侧导航入口或新数据库表，则立即升级为 **C 型**，停止按“简单节点补全”继续推进。
3. “简单节点补全”的最低完成标准必须同时满足：
   - 有真实执行能力
   - 有 `config_schema`
   - 有节点内配置入口
   - 有画布预览
   - 有执行面板 `compact` 摘要
   - 有错误双视图

## 0.1 真实代码基线

Prompt 装配以真实代码为准：

- 后端统一基线：`backend/app/nodes/_base.py`
- 拼接方式：`prompts/identity.md + nodes/_base_prompt.md + 各节点目录 prompt.md`

禁止再按旧草案写 `prompt_loader.py + prompts/nodes/*.md` 的实现说明。

## 1. 三种类型定义

### 1.1 A 型 — LLM 提示词节点

满足所有以下条件：

- 节点全部能力来自 `prompt.md` + LLM 调用
- 不需要调用任何外部 API 或第三方服务
- 后端文件数量：`__init__.py` + `node.py` + `prompt.md`，共 3 个文件以内
- 前端渲染：复用 `MarkdownRenderer` 或 `JsonRenderer`，或新增少量渲染器

细分为两个子类型：

| 子类型 | 特征 | 典型节点 |
|--------|------|---------|
| **A1 — 用户自选模型** | `requiresModel: true`，前端显示模型选择器 | `summary`、`flashcard`、`outline_gen` |
| **A2 — 系统指定模型** | `requiresModel: false`，前端不显示模型选择器，路由固定 | `ai_analyzer`、`ai_planner`（规划链路有特定路由要求） |

**走本文件夹下** → [`A型-LLM提示词节点-SOP.md`](./A型-LLM提示词节点-SOP.md)

---

### 1.2 B 型 — 外部工具节点

满足以下任一条件：

- 需要调用第三方 HTTP API（搜索引擎、OCR、翻译、支付等）
- 需要调用文件处理库（pypdf、python-docx、pandoc 等）
- `is_llm_node = False`，或 LLM 只是该节点工作流的一部分
- 需要新增环境变量（API Key 等）
- 可能需要新建 `services/xxx_service.py`

文件数量估算：4 ~ 8 个。

典型节点：`web_search`、`export_file`、`knowledge_base`、`write_db`

**走本文件夹下** → [`B型-外部工具节点-SOP.md`](./B型-外部工具节点-SOP.md)

---

### 1.3 C 型 — 插件（Plugin）

满足以下任意一项：

- 需要在**左侧导航栏**（`Sidebar.tsx`）独立占位一个入口
- 拥有独立的**前端功能页面**（不只是节点渲染器，而是完整 feature 模块）
- 需要新建 **Supabase 数据库表**（`supabase/migrations/` 新文件）
- 需要新建**后端 API 路由文件**（`backend/app/api/xxx.py`）
- 改动文件总数超过 10 个

文件数量估算：10 ~ 30+ 个。

典型例子（未来场景）：Anki 导出器、Notion 同步、PDF 批量处理器、代码审查助手

工作流节点只是插件的一个"触发点"或"联动接口"，不是插件的全部。

**走本文件夹下** → [`C型-插件-SOP.md`](./C型-插件-SOP.md)

## 2. 快速判断决策树

```
你要新增的功能是什么？
│
├── 只需写 prompt.md 和 node.py，调用 LLM 即可完成？
│   ├── 是 → A 型
│   │   ├── 用户可以选择模型？ → A1（用户自选模型）
│   │   └── 系统内部固定模型？ → A2（系统指定模型）
│   └── 否 ↓
│
├── 需要调用外部 API / 处理文件 / 依赖第三方服务？
│   ├── 是，但不需要侧边栏入口，不需要新 DB 表 → B 型
│   └── 否 ↓
│
└── 需要侧边栏入口 / 独立前端页面 / 新 DB 表中的任一项？
    └── 是 → C 型（插件）
```

## 3. 混合情况处理规则

### 3.1 B 型节点内部用到 LLM

例如：先调 Tavily API 搜索，再把结果喂给 LLM 做总结。

→ 仍然是 **B 型**，因为外部 API 是核心复杂点。
→ 节点的 `is_llm_node` 根据实际情况设定（混合节点建议 `True`）。
→ 对 LLM 路由部分，按 A 型规范配置 `task_routes`。

### 3.2 C 型插件内部包含工作流节点

例如：知识库插件包含 `knowledge_base` 节点（B 型）。

→ **先按 C 型 SOP 完成插件骨架**。
→ 节点部分在 C 型 SOP 中有专门的嵌套 Checklist，合并引用 A/B 型规范。

### 3.3 不确定是 A 型还是 B 型

判断标准：**节点的核心价值来自哪里？**

- 来自 Prompt 设计 → A 型
- 来自外部数据/服务 → B 型

## 4. 命名规范

所有新增节点类型字符串（`node_type`）必须：

- 使用 `snake_case`
- 表达具体功能，不用通用词（禁止 `tool`、`helper`、`util`）
- 与 `config.yaml` 中的键名完全一致
- 与前端 `NodeType` union 中的成员完全一致

示例：

```python
# ✅ 正确
node_type = "quiz_gen"
node_type = "mind_map"
node_type = "content_extract"

# ❌ 错误
node_type = "tool"
node_type = "myNode"
node_type = "LLMHelper"
```
