## 1. 概述

本文档定义 StudySolo 项目中各模块之间的边界、接口规范和依赖规则，确保团队协作时减少冲突、提高代码可维护性。

## 2. 前端模块边界

> 前端工程实践细则（目录约束、行数治理、Manifest-First、事件通信）见 [前端工程规范](08-前端工程规范.md)。
> 本节聚焦模块边界规则和依赖方向。

### 2.1 Feature Module 边界

**规则**：
1. Feature 模块是独立开发单元，有明确的 owner
2. Feature 之间通过 public API 通信，不直接导入内部组件
3. 共享组件必须移到 `components/ui/` 或 `features/admin/shared/`

**当前 Feature 模块**：

| 模块 | 路径 | Owner | 职责 |
|------|------|-------|------|
| workflow | `features/workflow/` | 羽升 | 工作流编辑器核心 |
| admin | `features/admin/` | 羽升 | 管理后台 |
| auth | `features/auth/` | 羽升 | 认证相关 |
| knowledge | `features/knowledge/` | 待定 | 知识库 |
| settings | `features/settings/` | 待定 | 用户设置 |

### 2.2 Store 边界

**规则**：
1. Store 只管理状态，不直接执行网络请求
2. Store 之间不允许隐式耦合（不能在一个 store 里直接调用另一个 store 的 state）
3. 跨 store 同步必须在调用方（hook 或组件）显式处理

**当前 Store（Phase 3 重组后）**：

| Store | 路径 | 职责 |
|-------|------|------|
| use-workflow-store | `stores/workflow/` | 工作流状态 + 执行 + 历史 |
| use-ai-chat-store | `stores/chat/` | AI 聊天 UI 状态 |
| use-conversation-store | `stores/chat/` | 对话历史持久化 |
| use-panel-store | `stores/ui/` | 面板开关状态 |
| use-settings-store | `stores/ui/` | 主题和偏好 |
| use-admin-store | `stores/admin/` | Admin sidebar |

**跨 store 耦合问题**：

| 问题 | 状态 | 说明 |
|------|------|------|
| `useAIChatStore.pushMessage` 直接调用 `useConversationStore.getState()` | ✅ 已修复（Phase 3） | 跨 store 同步改为在调用方显式处理 |
| 根级 `stores/*.ts` compat shim | ✅ 保留策略 | 向后兼容，不删除 |

### 2.3 Service 层边界

**规则**：
1. 所有 API 调用必须通过 `services/` 层
2. 使用 `api-client.ts` 提供的 `authedFetch` 而非裸 `fetch`
3. Service 之间不能循环依赖

**当前 Service**：

| Service | 路径 | 职责 |
|---------|------|------|
| api-client | `services/` | 统一 fetch + 认证 + 错误处理 |
| workflow.service | `services/` | 工作流 CRUD |
| auth.service | `services/` | 认证 API |
| admin.service | `services/` | Admin API |

**问题状态**：

| 问题 | 状态 | 说明 |
|------|------|------|
| `workflow.server.service.ts` 重复实现 `workflow.service.ts` 的部分函数 | ✅ 已显著收薄（Phase 3） | 主要重复已消除 |

### 2.4 前端事件通信（Phase 3 新增）

**TypedEventBus**（`frontend/src/lib/events/event-bus.ts`）：

```typescript
// 新代码必须使用 TypedEventBus
const unsubscribe = eventBus.on('workflow:execution-complete', (data) => {
  // ...
});
// 组件卸载时调用 unsubscribe()
```

**迁移策略**：
- 新代码必须用 `eventBus`
- 旧 CustomEvent 在涉及该文件的 PR 中逐步替换

## 3. 后端模块边界

### 3.1 API 路由边界

**规则**：
1. API 路由只做入参校验 + 权限控制，不承载长业务逻辑
2. 业务逻辑下沉到 `services/` 层
3. 按 domain 分组，避免碎片化

**当前路由结构（Phase 2 重组后）**：

| 路由包 | 路径 | 说明 |
|--------|------|------|
| auth | `api/auth/` | 认证路由包 |
| workflow | `api/workflow/` | 工作流路由包 |
| ai | `api/ai/` | AI 路由包 |
| admin | `api/admin/` | 管理后台路由包 |

**问题状态**：

| 问题 | 状态 | 说明 |
|------|------|------|
| workflow 路由碎片化（5 个散文件） | ✅ 已修复（Phase 2） | 合并为 `api/workflow/` 子目录 |
| `/ai` 前缀碎片化（5 个散文件） | ✅ 已修复（Phase 2） | 合并为 `api/ai/` 子目录 |
| ai_chat + ai_chat_stream 重复 | ✅ 已修复（Phase 2） | 合并为 `api/ai/chat.py` |

### 3.2 Service 层边界

**规则**：
1. Service 是独立业务逻辑单元
2. Service 只能调用下层（database, external API），不能调用 API 层
3. 共享逻辑提取到公共 Service

**当前 Service（Phase 2 重构后）**：

| Service | 路径 | 职责 |
|---------|------|------|
| llm/router | `services/llm/` | AI 多平台路由选择 |
| llm/caller | `services/llm/` | HTTP 调用 |
| llm/provider | `services/llm/` | provider client 管理 |
| llm/generators | `services/llm/generators/` | 流式/结构化生成器 |
| ai_chat | `services/ai_chat/` | AI 聊天共享逻辑（Phase 2 新增） |
| usage_ledger | `services/usage_ledger.py` | Usage 记录落账 |
| usage_analytics | `services/usage_analytics.py` | Usage 统计查询 |
| knowledge_service | `services/knowledge_service.py` | 知识库 facade |

**问题状态**：

| 问题 | 状态 | 说明 |
|------|------|------|
| ai_router 包含流式逻辑 | ✅ 已修复（Phase 2） | 流式提取到 `llm/generators/streaming.py` |
| usage_ledger 模式在多个 API 文件重复 | ✅ 已修复（Phase 2） | 提取为 `@track_usage` 装饰器 |

### 3.3 节点模块边界

**规则**：
1. 节点是独立插件，继承 `BaseNode`
2. 节点只能通过 `BaseNode` 定义的接口与引擎交互
3. 节点配置通过 `node_config` 传递，不硬编码

**当前节点结构**：

```
backend/app/nodes/
├── _base.py              # BaseNode 抽象基类（运行时权威）
├── _categories.py         # 节点分类枚举
├── _mixins.py            # 可复用 Mixin
├── __init__.py           # 节点注册表
├── input/                # 输入节点
├── analysis/             # 分析/控制节点
├── generation/           # 内容生成节点
├── interaction/          # 交互节点
├── output/               # 输出节点
└── community/            # 社区节点
```

**Manifest-First 原则（Phase 3 完成）**：
- 后端 manifest 是节点定义的唯一事实源
- 前端 `RENDERER_REGISTRY` 从 manifest 动态读取
- `workflow-meta.ts` 仍承担 `status / icon / theme / inputs / outputs` 结构职责

## 4. 模块间通信协议

### 4.1 前端 → 后端

**协议：REST API + JSON**

```typescript
// 请求
fetch('/api/workflow', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`,
  },
  body: JSON.stringify(data),
});

// 响应
{
  "data": {...},
  "error"?: string,
}
```

### 4.2 后端 → 子后端 Agent

**协议：OpenAI 兼容 HTTP API**

```
POST /v1/chat/completions
Content-Type: application/json
Authorization: Bearer <api-key>
X-Request-Id: <uuid>

{
  "model": "agent-name",
  "messages": [...],
  "stream": false
}
```

**响应**：
```json
{
  "id": "chatcmpl-xxx",
  "object": "chat.completion",
  "choices": [{
    "message": {"role": "assistant", "content": "..."}
  }]
}
```

## 5. 依赖规则

### 5.1 前端依赖方向

```
components/ui/  ←  所有模块可依赖
stores/         ←  components/ 和 hooks/ 可依赖
hooks/          ←  components/ 可依赖
features/       ←  不能相互依赖（除非通过 public API）
services/        ←  features/ 可依赖
types/           ←  所有模块可依赖（只读）
```

### 5.2 后端依赖方向

```
models/         ←  所有层可依赖（只读数据模型）
core/           ←  api/, services/, engine/ 可依赖
services/        ←  api/ 可依赖（api → services → models）
api/            ←  禁止依赖 engine/（防止循环）
engine/         ←  nodes/ 可依赖
nodes/          ←  禁止依赖 api/, services/
middleware/      ←  只被 main.py 引用
```

## 6. 新增模块流程

### 6.1 前端新增 Feature

1. 在 `features/` 下创建目录
2. 创建 `index.ts` 导出 public API
3. 在 `features/*/index.ts` 中声明 public 接口
4. 更新本文档的模块表

### 6.2 后端新增 Service

1. 在 `services/` 下创建文件
2. 在 `services/__init__.py` 导出（如果需要）
3. 在 `api/` 相应路由中使用
4. 更新本文档的模块表

### 6.3 新增节点

1. 在 `backend/app/nodes/<category>/` 下创建
2. 继承 `BaseNode`，设置 `node_type`
3. 注册到 `NODE_REGISTRY`（`__init__.py`）
4. 前端在 `RENDERER_REGISTRY` 添加对应渲染器

### 6.4 新增 Agent

1. 复制 `agents/_template` 为新目录
2. 实现 `src/core/agent.py` 的 Agent 逻辑
3. 运行四层契约测试
4. 在 `backend/config/agents.yaml` 中注册
5. 创建 PR

## 7. 违规处理

| 违规类型 | 处理方式 |
|---------|----------|
| Feature 间直接导入 | Code review 拒绝，要求通过 public API |
| Store 隐式耦合 | 标记为 tech debt，限期修复 |
| API 层调用 Service 外的逻辑 | Code review 拒绝 |
| 循环依赖 | CI 失败，必须解决 |
| 新代码使用 CustomEvent 而非 TypedEventBus | Code review 拒绝（Phase 3 后） |
