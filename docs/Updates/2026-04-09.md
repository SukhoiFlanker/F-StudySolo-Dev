# 2026-04-09 更新日志

## 1. StudySolo 会员阶梯额度系统实装 (Tier-Based Quota Hardening)

今日完成了 StudySolo 平台在「AI 对话」与「工作流执行」两大核心资源访问上的持久化数据库配额限制。这是继安全审计报告后的核心安全性补强动作，确保了符合 OWASP 2025 资源防枯竭标准的平台稳定性。

### 核心特性
1. **配额服务单一事实源 (`quota_service.py`)**
   - 确立了统一的会员额度表：
     - **Free**: 100次对话/天，20次执行/天，5步循环深度
     - **Pro**: 200次对话/天，50次执行/天，20步循环深度
     - **Pro+**: 500次对话/天，150次执行/天，100步循环深度
     - **Ultra**: 2000次对话/天，500次执行/天，无限制循环
   - 实现了基于**北京时间 (CST 0:00)**的重置逻辑。

2. **平滑降级（Soft Block）机制：AI 对话**
   - AI 聊天不再采取超出即阻断的生硬做法。
   - 超出日对话额度后，系统将自动将模型降级至基础模型 **Qwen Turbo**，并通过 SSE 通道下发 `quota_warning` 提示，保障用户的核心交流不断联。

3. **强阻断（Hard Block）机制：工作流执行**
   - 在高昂算力支出的工作流触发处（`workflow_execute.py`），对 `POST /api/workflow/{id}/execute` 实现了强阻断。
   - 当超过日执行上限后，返回带结构化错误体（包括当前用量及上限）的 `429 Too Many Requests`，前端可借此弹出明确提示。

4. **前后端接口扩展**
   - 扩展 `/api/usage/quota` 响应，不仅包含原本的「工作流保存数量」，还追加了全链路用量统计 `daily_chat_used` / `daily_execution_used`。

### 遗留技术债
* **并发执行数量限制**：现因历史脏数据问题导致后台不活跃的 SSE 幽灵运行记录积压（共 16 条无效 `running`）。并发防刷需在独立架构 Issue 追踪并解决数据一致性后方可开启。
