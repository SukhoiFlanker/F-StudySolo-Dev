## 一、项目全景概述

### 1.1 项目定位与愿景

StudySolo 是一个 **AI 驱动的学习工作流平台**。核心理念是：

> **一句话 → 一个完整的学习工作流**

用户通过自然语言描述学习目标（如"深入学习 React Hooks 的知识体系"），平台自动将其拆解为结构化的学习步骤节点，依次调用 AI 完成各个学习环节 —— 从大纲生成、知识精炼、重点提取到输出归档，形成**完整的学习记忆闭环**。

### 1.2 核心价值主张

| 维度 | 传统学习方式 | StudySolo 方式 |
|------|------------|---------------|
| **任务拆解** | 手动规划、经验依赖 | AI 自动拆解为结构化步骤 |
| **知识生成** | 逐一搜索、手动整理 | 多步骤 AI 链式生成 |
| **过程可视** | 纯文本笔记、线性记录 | 可视化工作流画布，实时看 AI 思考过程 |
| **记忆归档** | 分散保存、难以检索 | 语义向量检索 + 结构化存储 |
| **复用共享** | 个人独享 | 模板广场、一键复用 |
| **人机协同** | AI 一次性黑盒输出 | 随时暂停、随时干预、随时恢复 |

### 1.3 目标用户画像

| 用户类型 | 场景 | 核心诉求 |
|---------|------|---------| 
| **大学生** | 期末复习、课程预习 | 快速生成知识框架 + 重点提炼 |
| **考研/考证者** | 系统性备考 | 知识体系梳理 + 记忆归档 |
| **终身学习者** | 新技能/新领域入门 | 学习路径规划 + 结构化输出 |
| **内容创作者** | 知识输出 | 素材收集 + 大纲生成 + 内容润色 |

### 1.4 MVP 核心定义

> **MVP = 输入一句话 → AI 自动拆解为工作流节点 → 逐节点执行 AI → 流式展示结果**

**MVP 必须包含（P0）：**

| # | 功能 | 说明 |
|---|------|------|
| 1 | 用户认证 | 注册/登录（Supabase Auth + 邮箱验证码），Cookie HttpOnly 会话 |
| 2 | 工作流画布 | 基于 @xyflow/react，可视化展示节点和连线 |
| 3 | 自然语言→工作流生成 | 用户输入学习目标 → AI 分析拆解为多个步骤节点 |
| 4 | 工作流顺序执行 | 逐节点调用 AI，SSE 流式输出到画布节点内 |
| 5 | 工作流保存/加载 | 自动保存到 Supabase，支持继续编辑 |
| 6 | 基础安全 | CORS + Pydantic 验证 + JWT 鉴权 |

**MVP 不包含（后延）：**
模板广场（P1）、提示词管理系统（P1）、管理后台（P2）、Wiki 文档页（P2）、归档记忆系统（P2）、数据分析面板（P2）、复杂安全限流（P1）

## 二、核心技术选型与决策

> 💡 详细版本号、配置、部署方案见 → `PROJECT_PLAN.md`

### 2.1 架构总览

```
用户浏览器
    │
    ▼
阿里云 ECS（2核4G · 宝塔面板）
    │
    ├── Nginx 反向代理
    │     ├── /      → Next.js 前端 (port 2037)
    │     └── /api/  → FastAPI 后端 (port 2038)
    │
    └── 外部服务
          ├── Supabase Pro（PostgreSQL + Auth + pgvector）
          ├── 火山引擎 doubao-2.0-pro（简单任务 · 免费池）
          └── 阿里云百炼 qwen3-turbo（复杂任务）
```

### 2.2 关键技术决策

| 决策点 | 选择 | 核心理由 |
|--------|------|---------|
| **前端框架** | Next.js 16.1 + React 19.2 | App Router · SSR/SSG/ISR · Turbopack 稳定 |
| **后端框架** | FastAPI (Python 3.11+) | AI 生态最佳宿主语言 · ASGI 异步 · Pydantic V2 自动验证 |
| **工作流画布** | @xyflow/react 12.x | React 19 原生支持 · 拖拽/连线/缩放全家桶 |
| **状态管理** | Zustand 5.x | 轻量 · 极简 API · 配合 localforage 实现离线缓存 |
| **数据库** | Supabase Pro | PostgreSQL + RLS 行级安全 + pgvector 向量搜索 + Auth |
| **AI SDK** | openai Python SDK | 百炼 + 火山均兼容 OpenAI 格式，一个 SDK 通吃 |
| **部署** | 阿里云 ECS + 宝塔 | 完全控制、无冷启动、PM2 常驻、学习价值高 |
| **UI 组件** | Shadcn/UI + Tailwind v4.1 | 可访问性预优化 · OKLCH 色彩空间 · CSS-first |
| **前后端分离** | 独立进程 · Nginx 反代 | 独立管理 · 故障隔离 · 安全性（后端端口不暴露）|
| **配置管理** | YAML 配置中心化 + pydantic-settings | 模型/节点/容灾配置集中 config.yaml，零硬编码 |
| **Markdown 渲染** | react-markdown + remark-gfm + shiki + streamdown | AI 节点输出实时渲染 · SSE 增量解析 · 代码高亮 · 数学公式 |

### 2.3 前后端分离的核心优势

| 优势 | 说明 |
|------|------|
| **独立管理** | 前后端可独立重启、更新、扩展 |
| **故障隔离** | 后端 API 崩溃不影响前端页面展示 |
| **性能优化** | Nginx 直接处理静态资源，减轻应用压力 |
| **安全性** | 后端端口不对外暴露，仅通过 Nginx 反代访问 |

### 3.2 M2 用户认证系统

**认证方案**：基于 HttpOnly Cookie 的 JWT / Session（大厂标准方案）

**核心设计思路**：
- Supabase Auth 颁发 `access_token` + `refresh_token`，存储在 HttpOnly + Secure + SameSite=Lax 的 Cookie 中（**绝不使用 localStorage**）
- Next.js `middleware.ts` 边缘级路由守卫：未登录 → 重定向到登录页
- 跨 Tab 状态同步：`supabase.auth.onAuthStateChange` 监听登出事件
- 懒加载策略：先拉工作流列表元数据，点击后才加载完整 nodes/edges JSON

**认证能力**：
- 注册（邮箱+密码，触发 DirectMail 验证码）
- 登录 / 退出 / 无感刷新 Token / 获取当前用户信息

> 📌 深度实施方案见 → `daily_plan/user_auth/01-auth-and-guard-strategy.md`

### 3.3 M3 工作流画布

**画布核心**：@xyflow/react 12.x · 可视化节点+连线+拖拽

**同步策略 — 三层防抖**：
1. **UI 层**（0 延迟）：React Flow 操作实时同步到 Zustand 并重绘
2. **本地存储层**（500ms 防抖）：写入 IndexedDB（localforage），附带脏数据标记
3. **云端同步层**（3-5s 防抖）：PUT 到 FastAPI → Supabase

**异常恢复（断网/崩溃）**：
- IndexedDB 存储本地更新时间戳，与云端时间戳对比
- 若本地更新 > 云端 → 弹窗提示恢复未保存数据

**工作流 CRUD**：
- 列表查询（仅元数据，不含完整 JSON）
- 新建 / 获取详情 / 更新（自动保存）/ 删除

> 📌 深度实施方案见 → `daily_plan/workflow_canvas/01-editor-autosave-sync-strategy.md`

### 3.4 M4+M5 AI 引擎 + 工作流执行

**自然语言→工作流生成**：
- 接收用户输入的学习目标
- AI 分析拆解为 3-8 个步骤
- 返回结构化 JSON：`{ nodes: [], edges: [] }`（直接可用于 React Flow 渲染）

**节点类型体系**（MVP 核心 4 种 + 扩展）：
- `outline_gen` — 大纲生成
- `content_extract` — 知识提炼
- `summary` — 总结归纳
- `flashcard` — 闪卡生成
- 每个节点有独立的 System Prompt 模板

**工作流执行引擎**：
- 按 edges 拓扑排序遍历节点
- 逐节点调用 AI（根据节点类型路由到不同模型）
- SSE 逐 token 流式推送到前端画布
- 节点状态生命周期：`Pending → Running → Done / Error`

> 📌 节点体系完整设计见 → `daily_plan/core/节点分析.md`  
> 📌 工作流交互设计见 → `daily_plan/core/工作流AI交互规划.md`

## 四、工作流核心引擎

> 本节从产品视角阐述工作流执行的核心机制，技术实现见 core 专项文档。

### 4.1 两段式 AI 生成

用户输入一句自然语言后，经过两段 AI 处理：

```
用户输入："帮我深入学习 React Hooks"
         │
         ▼
┌─────────────────────────────────┐
│  第一个 AI（需求分析器）          │
│  模型：豆包 2.0-pro（免费池）     │
│  任务：提炼需求、结构化 JSON 输出  │
│  输出：{目标、流程、要求、限制}    │
└─────────────┬───────────────────┘
              │
              ▼
┌─────────────────────────────────┐
│  第二个 AI（工作流规划器）         │
│  模型：qwen3-turbo（推理力强）    │
│  任务：从预设节点池中编排工作流     │
│  输出：{ nodes: [], edges: [] }  │
└─────────────┬───────────────────┘
              │
              ▼
         画布渲染 + 逐节点执行
```

### 4.2 用户可配置项

在输入区域旁边，用户可以调整：

| 配置项 | 选项 | 影响 |
|--------|------|------|
| **思考深度** | 低 / 中 / 高 | 第一个 AI 的分析精细度 |
| **工作流步数偏好** | 更少 / 默认 / 更多 | 第二个 AI 生成的节点数量 |
| **数据获取模式** | 知识库 / 联网搜索 / 混合 | 决定是否启用搜索/RAG 节点 |

### 4.3 执行与流式输出

```
引擎执行循环：
  for each node in topological_sort(edges):
    1. 注入暗线全局上下文到 System Prompt
    2. 根据 node.type 路由到对应模型
    3. SSE 流式推送每个 token 到前端
    4. 前端实时更新画布节点内容 + 状态动画
    5. 节点完成后，输出传递给下游节点
```

> 📌 节点 JSON Schema 定义、状态机、模型路由映射 → `daily_plan/core/节点分析.md`

## 五、AI 引擎与双模型路由

### 5.1 双模型路由策略

根据任务复杂度，自动将请求路由到最合适的模型：

```
用户请求
   │
   ├─ 简单任务（意图识别 · 节点拆解 · 短文分类 · 闪卡生成）
   │    └── 火山引擎 doubao-2.0-pro
   │         · 200W Token/日免费池 · 高并发 · 低延迟
   │
   ├─ 复杂任务（大纲生成 · 知识总结 · 长文润色 · 工作流编排）
   │    └── 阿里云百炼 qwen3-turbo
   │         · 高质量输出 · 复杂逻辑推理
   │
   └─ 容灾降级
        └── 任一侧 timeout/rate limit → 自动切换备用提供商
```

**统一调用**：两家均兼容 OpenAI API 格式，使用 `openai` Python SDK 统一调用，零适配成本。

### 5.2 模型能力按等级解锁

| 用户等级 | 可用模型范围 | 策略 |
|---------|------------|------|
| **免费版** | 仅国内基础模型（豆包 + 千问） | 成本控制 |
| **Pro 版** | 满血模型（含 GPT-4、Claude-3 等） | 体验升级 |
| **学生认证** | 免费版也可调用满血模型 | 教育普惠 |
| **Ultra 版** | 所有旗舰模型，最新版本优先 | 无限制 |

### 5.3 YAML 配置驱动

所有模型配置、节点→模型映射、容灾策略均集中在 `config.yaml` 中管理：

- **模型配置**：name / provider / base_url / api_key / max_tokens / temperature / route
- **节点配置**：每个节点类型绑定默认模型和参数
- **执行引擎**：最大并行数 / 重试次数 / 超时时间
- **容灾降级**：超时阈值 / 降级优先级

代码侧通过 `config_loader.py` 加载配置，`ai_router.py` 零硬编码调用 → 改模型/调参数只需修改 YAML，无需改代码。

> 📌 config.yaml 完整结构和代码示例 → `daily_plan/core/02-yaml-config-and-markdown-rendering.md`

### 5.4 AI 安全防护

- 用户输入严格转义（防 Prompt Injection）
- System Prompt 沙箱指令设计，限制 AI 行为边界
- API Key 仅存于后端环境变量，前端零泄露

### 5.5 四平台统一路由（增强版）

接入平台（优先级从高到低）：

| 优先级 | 平台 | 核心优势 |

## 六、知识库 RAG 智能检索体系

> **核心思想："不要让大模型每次都读全文"**  
> 构建"摘要 – 向量 – 原文"三层漏斗，在保证回答质量的前提下，将 token 消耗降低一个数量级。

### 6.1 三层漏斗架构

| 层级 | 功能 | 覆盖度 | Token 消耗 |
|------|------|--------|-----------|
| **摘要层** | 文档上传后 AI 自动生成核心观点、目录架构、深度摘要 | ~80% 日常问答 | 极低 |
| **向量层** | 文档分块→向量化→语义检索最相关片段 | ~15% 精准定位 | 中等 |
| **原文层** | 极少数需精确引用或深度分析时，读取完整原文 | ~5% 极端需求 | 高 |

### 6.2 文档预处理管道

用户上传文档（Word、PDF、Markdown、TXT、图片）后自动触发：

1. **文档解析** → 提取纯文本，按段落/标题切分成 chunk（500-800 tokens），保留上下文重叠
2. **AI 总结** → 生成摘要（500-1000字）、核心观点（列表）、目录架构（JSON，指向原文位置）
3. **向量化存储** → 摘要和分块原文分别向量化，存入 Supabase pgvector

### 6.3 问答检索三级策略

```
用户提问
    ▼
第一步：检索摘要层 → 能回答？→ 直接返回 ✅
    ▼ 不够用
第二步：深入向量层 → 找到最相似的 3-5 个原文片段 → 精确回答 ✅
    ▼ 极少数情况
第三步：定位原文 → 根据目录架构定位具体位置 → 返回完整段落 ✅
```

### 6.4 联网搜索模式

- **多源并行搜索**：权威源（学术、政府、新闻）+ 论坛源（知乎、技术社区），每个结果附来源标签
- **内容总结**：AI 生成搜索结果综合摘要，附上原始链接
- **两级问答**：先用摘要回答，需要时才展开原文

### 6.5 缓存与存储分层

| 层级 | 存储内容 | 存储技术 | 用途 |
|------|---------|---------|------|
| **文档原文** | 用户上传的原始文件 | 对象存储 | 极少访问，精确溯源 |
| **摘要层** | AI 生成的摘要/观点/目录 | Supabase 关系表 | 快速回答，结构化查询 |
| **向量层** | 摘要向量 + 原文分块向量 | pgvector | 语义检索 |
| **缓存层** | 高频查询的答案和上下文 | Redis / KV 存储 | 降低重复计算 |

### 6.6 知识库存储限制

| 限制项 | 规则 |
|--------|------|
| 每个知识库容量上限 | 100 MB |
| 用户知识库数量 | 无上限 |
| 总存储空间上限 | 1 GB（免费版），按等级递增 |
| 超出后处理 | 需付费升级（¥1/GB/月） |
| 支持格式 | Word、PDF、Markdown、TXT、图片 |

## 七、四大核心创举

> 这是 StudySolo 区别于市面上所有 AI 工具的核心竞争力。

### 7.1 明暗线双轨架构 (Explicit & Implicit Architecture)

**痛点**：传统线性工作流（A → B → C）中，大模型会"失忆"且"精神分裂"。不同节点生成的内容风格不统一、逻辑割裂。

**解决方案**：

| 维度 | 暗线（总编 · 全局上下文） | 明线（记者团队 · 任务执行者） |
|------|----------------------|--------------------------|
| **本质** | 驻留在内存/数据库中的 JSON 状态机 | 真正干活的单一节点 |
| **内容** | 全局主题、语言风格、核心大纲、受众定位 | 具体任务：写前言、写正文、写结论 |
| **生命周期** | 从工作流开始到任务结束，一直存在且可动态更新 | 执行完毕即释放 |
| **协同方式** | 系统自动将暗线数据注入每个明线节点的 System Prompt | 每个节点在"潜意识"里遵循同一个规则 |

**最终效果**：不管明线拆分成多少细碎任务，每一个 AI 节点都在统一的"总编审稿"下工作，输出风格一致、逻辑连贯。

### 7.2 动态打断与人机协同 (Human-in-the-Loop)

**痛点**：绝大多数 AI 工具一旦运行就像脱缰野马，用户只能干等几分钟看盲盒结果。

**三步人机协同机制**：

| 步骤 | 操作 | 系统行为 |
|------|------|---------|
| **暂停 (Pause)** | 用户点击停止 | 数据库状态秒变 `paused`，当前节点跑完后掐断后续 |
| **人工覆写 (Override)** | 用户直接修改 AI 输出 | 前端发更新指令，替换该节点的 Output 数据 |
| **满血复活 (Resume)** | 用户点击继续 | 引擎带着用户改过的正确数据继续执行下游节点 |

> 🎯 **核心理念**：AI 只是苦力，人才是真正的驾驶员。

**技术保障**：异步任务队列 + 数据库实时状态字段（`running` / `paused` / `failed`），绝不用死循环跑节点。

### 7.3 拆解聚合与格式化输出 (Map-Reduce & Export)

**痛点**：大模型有单次输出字数限制（Token Limit），万字报告要么截断要么偷懒。

**Map-Reduce 模式**：

| 阶段 | 说明 |
|------|------|
| **Map（分拆）** | 系统根据暗线中的大纲，自动生成多个独立生成任务（前言/正文/结论），**可并行执行**，极大缩短等待时间 |
| **Reduce（聚合）** | 专门的"润色合并节点" —— 不产生新观点，只梳理上下文衔接、消除重复话术、统一风格，使其变成连贯长文 |
| **Export（实体化导出）** | 合并后的 Markdown → 一键转换为排版精美的 PDF / DOCX，前端直接下载 |

### 7.4 可观测性：透明化运行看板 (Observability UI)

**目标**：建立用户信任，区分"黑盒工具"与"专业系统"。不仅给程序员看 Log，更要让普通用户看懂 AI 的思考过程。

**三个核心能力**：

| 能力 | 实现 |
|------|------|
| **呼吸感 UI** | WebSocket 长连接，后端每执行一步推一条消息 → 前端节点亮呼吸灯 → 完成打绿色对勾 |
| **数据脱水与标签化** | 原始 JSON 自动转译为友好标签。如 `{"keywords": ["生物医药"]}` → `🔍 正在搜索：生物医药` |
| **Markdown 实时预览** | 节点产出文章片段时，右侧边栏像打字机一样实时渲染带格式的 Markdown 内容（react-markdown + streamdown 增量解析，无闪烁） |

> 📌 四大核心创举的完整技术方案见 → `daily_plan/core/工作流AI交互规划.md` §四~§七

## 八、数据模型设计思路

### 8.1 核心表设计

| 表名 | MVP | 功能定位 |
|------|-----|---------|
| `users` | ✅ | 用户账户（Supabase Auth 自动管理） |
| `workflows` | ✅ | 工作流元数据 + nodes/edges JSONB 画布数据 |
| `workflow_runs` | ✅ | 执行历史记录 + tokens 消耗统计 |
| `prompts` | ❌ P1 | 提示词管理 + 版本控制 |
| `prompt_versions` | ❌ P1 | 提示词版本历史（支持一键回滚） |
| `workflow_templates` | ❌ P1 | 工作流模板广场（共享 + 收藏 + 排行） |
| `memories` | ❌ P2 | 归档记忆 + pgvector 语义向量搜索 |

### 8.2 安全策略

- 所有表均开启 **Row Level Security（RLS）**
- 即便伪造 API 请求，JWT 中的 UID 不匹配则数据库拒绝操作
- 用户只能访问自己的数据，从数据库层面彻底杜绝越权

### 8.3 存储策略

- 工作流画布数据以 **JSONB** 格式存储 nodes/edges，兼顾灵活性和查询性能
- 工作流列表接口只返回元数据（ID、名称、更新时间），**不返回完整 JSON**
- 用户点击具体工作流时才懒加载 nodes/edges → 节省带宽和首屏时间
- 本地 IndexedDB 作为离线缓存层 → 断网/崩溃后可恢复

| 组件 | 内存占用 |
|------|---------|
| 系统 + 宝塔面板 | ~800 MB |
| Nginx | ~50 MB |

- 不在服务器自建数据库（Supabase 云释放资源）
