## 一、项目全景概述

### 1.1 项目定位与愿景

StudySolo 是一个 **AI 驱动的学习工作流平台**。核心理念是：

> **一句话 → 一个完整的学习工作流**

用户通过自然语言描述学习目标（如"深入学习 React Hooks 的知识体系"），平台自动将其拆解为结构化的学习步骤节点，依次调用 AI 完成各个学习环节——从大纲生成、知识精炼、重点提取到输出归档，形成**完整的学习记忆闭环**。

### 1.2 核心价值主张

| 维度 | 传统学习方式 | StudySolo 方式 |
|------|------------|---------------|
| **任务拆解** | 手动规划、经验依赖 | AI 自动拆解为结构化步骤 |
| **知识生成** | 逐一搜索、手动整理 | 多步骤 AI 链式生成 |
| **过程可视** | 纯文本笔记、线性记录 | 可视化工作流画布 |
| **记忆归档** | 分散保存、难以检索 | 语义向量检索 + 结构化存储 |
| **复用共享** | 个人独享 | 模板广场、一键复用 |

### 1.3 目标用户画像

| 用户类型 | 场景 | 核心诉求 |
|---------|------|---------|
| **大学生** | 期末复习、课程预习 | 快速生成知识框架 + 重点提炼 |
| **考研/考证者** | 系统性备考 | 知识体系梳理 + 记忆归档 |
| **终身学习者** | 新技能/新领域入门 | 学习路径规划 + 结构化输出 |
| **内容创作者** | 知识输出 | 素材收集 + 大纲生成 + 内容润色 |

### 1.4 MVP 核心定义

> **MVP = 输入一句话 → AI 自动拆解为工作流节点 → 逐节点执行 AI → 流式展示结果**

**MVP 必须包含（P0）：**

| # | 功能 | 说明 |
|---|------|------|
| 1 | 用户认证 | 注册/登录（Supabase Auth + 邮箱验证码），Cookie HttpOnly 会话 |
| 2 | 工作流画布 | 基于 @xyflow/react，可视化展示节点和连线 |
| 3 | 自然语言→工作流生成 | 用户输入学习目标 → AI 分析拆解为多个步骤节点 |
| 4 | 工作流顺序执行 | 逐节点调用 AI，SSE 流式输出到画布节点内 |
| 5 | 工作流保存/加载 | 自动保存到 Supabase，支持继续编辑 |
| 6 | 基础安全 | CORS + Pydantic 验证 + JWT 鉴权 |

**MVP 不包含（后延）：**
模板广场（P1）、提示词管理系统（P1）、管理后台（P2）、Wiki 文档页（P2）、归档记忆系统（P2）、数据分析面板（P2）、复杂安全限流（P1）

### 3.2 M2 用户认证系统（深度设计）

**认证方案**：基于 HttpOnly Cookie 的 JWT / Session（大厂标准方案）

**核心流程**：
- Supabase Auth 颁发 `access_token` + `refresh_token`
- Token 存储在 `HttpOnly` + `Secure` + `SameSite=Lax` 的 Cookie 中（**绝不使用 localStorage**）
- `@supabase/ssr` 开箱即用管理 Cookie 会话

**路由守卫**：Next.js `middleware.ts` 边缘级拦截
- 受保护路由：`/workspace`、`/settings`、`/history`
- 未登录 → 重定向到 `/login?next={pathname}`

**跨 Tab 状态同步**：
- `supabase.auth.onAuthStateChange` 监听登出事件
- `localStorage` 写入 `auth_event` 触发其他 Tab 同步跳转

**懒加载数据拉取**：
1. Server Component 先拉取工作流列表元数据（ID、名称、更新时间）
2. 用户点击工作流 → Client Component 用 React Query 拉取完整 nodes/edges JSON
3. JSON 数据存入 Zustand + localforage（IndexedDB）

**后端路由**：
- `POST /api/auth/register` — 注册（邮箱+密码，触发 DirectMail 验证码）
- `POST /api/auth/login` — 登录（返回 access_token + refresh_token）
- `POST /api/auth/logout` — 退出
- `POST /api/auth/refresh` — 无感刷新 Token
- `GET /api/auth/me` — 获取当前用户信息（JWT 鉴权）

### 3.3 M3 工作流画布（深度设计）

**画布核心**：@xyflow/react 12.x

**三层防抖同步策略**：
1. **UI 层**（0 延迟）：React Flow 操作实时同步到 Zustand 并重绘
2. **本地存储层**（500ms 防抖）：写入 IndexedDB（localforage），附带 `dirty=true` 时间戳
3. **云端同步层**（3-5s 防抖）：PUT 请求到 FastAPI Supabase

**离开拦截**：
- `window.onbeforeunload` + Zustand `isDirty` 脏数据标记
- Next.js Router 拦截器 → 自定义弹窗询问"是否保存"

**异常恢复（断网/崩溃）**：
1. IndexedDB 存储 `local_updated_at` 时间戳
2. 云端返回 `cloud_updated_at` 时间戳
3. 下次加载时比较：若 `local_updated_at > cloud_updated_at` → 弹窗提示恢复

**后端路由**：
- `GET /api/workflow/` — 获取工作流列表（元数据，不含 nodes/edges）
- `POST /api/workflow/` — 新建工作流
- `GET /api/workflow/{id}/content` — 获取完整 nodes/edges JSON
- `PUT /api/workflow/{id}` — 更新工作流（自动保存）
- `DELETE /api/workflow/{id}` — 删除工作流

### 3.4 M4+M5 AI 引擎 + 工作流执行（深度设计）

**自然语言→工作流生成**（`/api/ai/generate-workflow`）：
- 接收用户输入的学习目标
- AI 分析拆解为 3-8 个步骤
- 返回结构化 JSON：`{ nodes: [], edges: [] }`（直接可用于 React Flow）

**节点类型体系**：
- `outline_gen` — 大纲生成
- `content_extract` — 知识提炼
- `summary` — 总结归纳
- `flashcard` — 卡片生成
- 每个节点有独立的 System Prompt 模板

**工作流执行引擎**（`/api/workflow/{id}/execute`）：
- 按 edges 顺序遍历节点
- 逐节点调用 AI（根据节点类型路由到不同模型）
- SSE（`sse-starlette`）逐 token 流式推送

**前端 SSE 接收**：
- `EventSource` 接收 SSE 流
- 逐 token 更新对应节点的 `data.output`（Zustand + React Flow 实时重绘）
- 节点状态：`Pending → Running → Done / Error`

## 四、数据模型与存储设计

### 4.1 核心表结构

| 表名 | MVP | 说明 |

| `users` | ✅ | Supabase Auth 自动管理 |
| `workflows` | ✅ | 工作流元数据 + nodes/edges JSONB |

### 4.2 RLS 安全策略

所有表均开启 Row Level Security（RLS），确保用户只能访问自己的数据。即便伪造 API 请求，JWT 中的 UID 不匹配则数据库拒绝操作。

```
  │
  └─ 容灾降级
       └── 任一侧 timeout/rate limit → 自动切换备用提供商
```

**统一调用**：两家均兼容 OpenAI API 格式，使用 `openai` Python SDK 统一调用。

### 5.2 安全防护

- 用户输入严格转义（防 Prompt Injection）
- System Prompt 沙箱指令设计
- API Key 仅存在后端 `.env`，前端零泄露

## 九、性能优化策略

### 9.1 2核4G 资源分配

```
系统 + 宝塔面板      ~800 MB
Nginx                ~50 MB
Next.js (PM2)        ~800 MB - 1.2 GB
FastAPI (Gunicorn)   ~300 MB - 500 MB
系统缓存/Buffer      ~1 GB
预留                  ~500 MB
─────────────────────────────
总计                  ~3.5 GB (有余量)
```

### 9.2 优化措施

- PM2 前端 1 实例 · 后端 Gunicorn 2 workers（利用双核 I/O 调度）
- 不在服务器自建数据库（Supabase 云释放资源）
- Swap 2GB 兜底 · Nginx Gzip/Brotli 压缩
- Next.js SSG/ISR 预渲染 · `next/image` 图片优化
- 静态资源 365 天长缓存
