## 一、节点的本质

AI 节点在工作流中的本质，就是一个**被严格约束的函数（Function）**。

在 Coze、Dify 或是 n8n 中，所谓的 AI 节点，底层逻辑就是通过 JSON Schema 强行规定：

- **输入（Input）：** 上游节点传来的变量（比如搜索结果）。
- **指令（System Prompt）：** AI 在这个节点扮演什么角色，要干什么活。
- **输出（Output Format）：** 强制 AI 以特定的 JSON 格式返回结果（比如 `{"title": "...", "summary": "..."}`），绝不允许它多说一句废话。只有这样，下游节点才能稳定地把它的结果拿去用。

## 二、4 大基础流派（宏观分类）

基于"简洁有力"的理念，深度拆解了目前市面上最顶级的平台（n8n、Make、Coze、Dify、ComfyUI），抛开花里胡哨的包装，从最大的宏观视角来看，所有的工作流节点，有且只有这 **4 大基础流派**（也就是 4 种功效）：

### 2.1 触发类节点 (Triggers) —— "引擎的点火器"

这类节点只有一个功效：**决定工作流在什么时候、因为什么原因开始运行。**

遵循"简单化"原则，不需要搞几十种触发器，核心就三种：

| 触发方式 | 说明 | 在 StudySolo 中的角色 |
|---------|------|--------------------|
| **对话/输入触发 (Chat/Form Input)** | 用户在对话框发了一句话，或者填了一个表单，立刻启动 | ⭐ **最核心的触发方式** |
| **定时触发 (Schedule/Cron)** | 比如"每天早上 8 点自动执行一次" | 🔮 后期扩展（如定时复习提醒） |
| **事件触发 (Webhook/Event)** | 监听外部信号，比如"当收到一封新邮件时" | 🔮 后期扩展（如知识库更新触发） |

### 2.2 外部感知节点 (Data & Context) —— "引擎的眼睛和耳朵"

这类节点不负责思考，只负责**无脑且精准地把外部数据搬运进来**，为后面的 AI 提供弹药。

| 感知方式 | 说明 | 在 StudySolo 中的角色 |
|---------|------|--------------------|
| **知识库/RAG 节点** | 去专属数据库里捞出相关的本地文档 | ⭐ 对应"纯净版知识库模式" |
| **联网搜索节点** | 带着关键词去互联网抓取最新的网页文本 | ⭐ 对应"联网搜索模式" |
| **基础读取节点 (Read API/DB)** | 读取特定的数据库记录，或调用外部 API | 🔮 后期扩展 |

> **本质原则：** 这类节点绝不能做复杂处理，查到什么就输出什么（纯文本或 JSON），**保持职责单一**。

### 2.3 大脑与中枢节点 (Processing & Logic) —— "引擎的齿轮与 CPU"

这是最核心的部分，也是体现"复杂问题拆解化"的地方。不要指望一个节点搞定所有事，而是用不同的逻辑节点组合。

| 逻辑类型 | 说明 | 在 StudySolo 中的角色 |
|---------|------|--------------------|
| **大模型节点 (LLM Node)** | 需要"理解力"的工作：总结、翻译、提取结构化数据 | ⭐ **占比最高的核心节点** |
| **逻辑控制节点 (If/Else & Switch)** | 分流判断，如"评分 >80 走 A 路线" | ⭐ 工作流分支控制 |
| **循环迭代节点 (Loop/Map)** | 对付重复劳动，如对 10 篇文章逐一总结 | ⭐ Map-Reduce 拆解聚合 |
| **代码/脚本节点 (Code/Custom)** | AI 容易出错的死板逻辑：日期转换、正则匹配 | ⚡ 自定义节点能力 |

### 2.4 动作与输出节点 (Actions) —— "引擎的手脚"

当数据处理完毕后，工作流必须对世界产生实质性影响，这类节点执行最终指令。

| 动作类型 | 说明 | 在 StudySolo 中的角色 |
|---------|------|--------------------|
| **回复用户节点 (Chat Response)** | 把渲染好的 Markdown/文本发回前端 | ⭐ 逐节点 SSE 流式输出 |
| **写入/更新节点 (Write DB/API)** | 把处理结果存入 Supabase/生成 PDF | ⭐ 归档记忆 + 导出功能 |

## 四、暗线机制与节点的关系

> 暗线（全局上下文管理器）不是一个可见的节点，而是一个贯穿所有节点的**隐形状态机**。  
> 详见 [工作流AI交互规划 · 明暗线双轨架构](./工作流AI交互规划.md#四核心创举明暗线双轨架构)

### 4.1 暗线数据结构

```json
{
  "implicit_context": {
    "global_theme": "XXX村产业调研",
    "language_style": "严谨的学术报告，多用客观数据",
    "core_outline": ["背景", "现状", "问题", "对策"],
    "target_audience": "高校评委",
    "user_constraints": {
      "max_steps": 6,
      "mode": "knowledge"
    }
  }
}
```

### 4.2 暗线注入机制

每个 **Processing (LLM)** 类节点在执行时，引擎会自动将暗线数据追加到该节点的 System Prompt 中：

```python
# workflow_engine.py 伪代码
async def execute_node(node, implicit_context):
    system_prompt = node.data.system_prompt
    # 暗线注入：强制全局一致性
    system_prompt += f"\n\n[GLOBAL CONTEXT - 必须遵循]\n{json.dumps(implicit_context, ensure_ascii=False)}"
    
    response = await ai_client.chat.completions.create(
        model=node.data.model_route,
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": json.dumps(node.input)}
        ],
        stream=True
    )
    # SSE 流式输出...
```

## 五、节点状态机

每个节点在工作流运行时，具有以下生命周期状态：

```
                    ┌──── 用户手动覆写 ────┐
                    │                      │
  pending → running → done ──────────────→ │
               │                           ▼
               └─→ error → retry → running (重新执行)
               │                       
               └─→ paused (人工介入) → running (满血复活)
```

| 状态 | 含义 | 前端 UI 表现 |
|------|------|-------------|
| `pending` | 等待执行 | 灰色节点，无动画 |
| `running` | 正在执行 | 呼吸灯动画，右侧 Markdown 实时预览 |
| `done` | 执行完成 | 绿色对勾 ✅ |
| `error` | 执行失败 | 红色叹号 ❌，显示错误信息 |
| `paused` | 人工暂停 | 橙色暂停图标 ⏸️ |

## 六、节点与模型路由映射

基于前文定义的双模型路由策略，每个节点类型绑定默认模型：

| 节点类型 | 默认模型 | 路由理由 | Token 消耗量级 |
|---------|---------|---------|--------------|
| `ai_analyzer` | 豆包 2.0-pro | 短文本分析，走免费池 | ~500-1K |
| `ai_planner` | qwen3-turbo | 需要复杂推理编排能力 | ~1K-2K |
| `outline_gen` | qwen3-turbo | 结构化长文本生成 | ~1K-3K |
| `content_extract` | qwen3-turbo | 深度知识提炼 | ~2K-5K |
| `summary` | qwen3-turbo | 多源综合归纳 | ~1K-3K |
| `flashcard` | 豆包 2.0-pro | 简单格式化任务 | ~300-800 |
| `merge_polish` | qwen3-turbo | 长文润色需推理力 | ~3K-8K |

> **容灾机制：** 当任一模型 timeout/rate limit 时，自动降级至备用提供商。详见 [项目深度功能规划 · AI 引擎深度设计](../../global/项目深度功能规划.md#五ai-引擎深度设计)。

## 七、具体场景示例：一次完整的节点分解

**用户输入：** "帮我深入学习 React Hooks 的知识体系，要系统全面，适合有基础的开发者"

### Step 1：`ai_analyzer` 输出

```json
{
  "goal": "系统学习 React Hooks 完整知识体系",
  "user_defined_steps": [],
  "design_requirements": "系统全面，适合有基础的开发者",
  "constraints": { "has_step_limit": false, "max_steps": null },
  "extras": "受众为有一定 React 基础的开发者"
}
```

### Step 2：`ai_planner` 输出（节点编排）

```json
{
  "nodes": [
    { "id": "n1", "type": "outline_gen", "data": { "label": "📋 生成 Hooks 知识大纲" } },
    { "id": "n2", "type": "content_extract", "data": { "label": "📖 基础 Hooks 深度提炼 (useState/useEffect)" } },
    { "id": "n3", "type": "content_extract", "data": { "label": "📖 进阶 Hooks 深度提炼 (useCallback/useMemo/useRef)" } },
    { "id": "n4", "type": "content_extract", "data": { "label": "📖 高级模式提炼 (自定义 Hooks/性能优化)" } },
    { "id": "n5", "type": "summary", "data": { "label": "📝 知识体系总结归纳" } },
    { "id": "n6", "type": "flashcard", "data": { "label": "🃏 生成记忆闪卡" } }
  ],
  "edges": [
    { "source": "n1", "target": "n2" },
    { "source": "n1", "target": "n3" },
    { "source": "n1", "target": "n4" },
    { "source": "n2", "target": "n5" },
    { "source": "n3", "target": "n5" },
    { "source": "n4", "target": "n5" },
    { "source": "n5", "target": "n6" }
  ]
}
```

### Step 3：执行流可视化

```
[outline_gen] ─┬─→ [content_extract: 基础] ──┐
               ├─→ [content_extract: 进阶] ──┤──→ [summary] ──→ [flashcard]
               └─→ [content_extract: 高级] ──┘
```

> 注意：n2/n3/n4 三个 `content_extract` 节点可以**并行执行**（Map 阶段），然后汇聚到 `summary`（Reduce 阶段），这就是 Map-Reduce 模式的实际落地。
