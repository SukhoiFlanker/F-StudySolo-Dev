#### 权益叠加与计费独立性

- **权益叠加**：用户可同时购买任意包月会员和按量加购包，所有权益**直接叠加**。例如Pro版用户加购+5GB存储后，总存储配额变为3GB+5GB=8GB；加购+5个工作流后，总工作流上限变为50+5=55个。
- **计费周期分别计算**：包月会员和每个加购项拥有独立的计费周期和自动续费设置。系统通过不同的数据库表记录（会员订阅表、加购项表），互不影响。
- **续费/取消独立**：用户可单独取消加购项而不影响主会员，反之亦然。主会员过期时，所有加购项同步失效（按保留期规则处理）；主会员续费后，加购项需重新购买或自动恢复（取决于过期时长）。

#### 加购自动续费逻辑

- 开通加购项时默认开启自动续费，每月扣款日前3天发送提醒。
- 扣款失败有3天宽限期，期间加购权益保留。
- 连续2个月扣款失败，自动取消加购项。
- 加购项取消后，对应的资源配额即时收回（如存储空间减少），但已存储的数据不受影响（仅当总使用量超过新配额时触发只读模式）。

## 二、核心工作流交互机制

> **核心流程概述：** 用户输入自然语言 → AI 分析器提炼需求 → AI 规划器生成工作流 → 逐节点执行 → 流式展示结果  
> 🔗 节点详细定义参见 → [节点分析](./节点分析.md)

### 2.1 第一阶段：用户输入与初始配置

用户输入一段自然语言后，在对话框中可以配置：

| 配置项 | 选项 | 说明 |
|--------|------|------|
| **思考深度** | 低 / 中 / 高 | 影响第一个 AI 的分析精细度 |
| **工作流步数偏好** | 更少 / 默认 / 更多 | 影响第二个 AI 生成的节点数量 |
| **数据获取模式** | 知识库 / 联网搜索 / 混合 | 决定是否启用搜索/RAG 节点 |

### 2.2 第二阶段：AI 分析器（`ai_analyzer`）

由**豆包 2.0-pro**（支持思考强度选择）深度分析用户需求，最终以 JSON 格式输出结构化结果：

```json
{
  "goal": "用户需求（工作流主题是什么）",
  "user_defined_steps": ["用户自己规定的工作流流程"],
  "design_requirements": "设计审美功能方面的要求",
  "constraints": {
    "has_step_limit": true,
    "max_steps": 5
  },
  "extras": "其他补充信息"
}
```

> 📌 此 JSON 即为 `ai_analyzer` 节点的标准输出格式，详见 [节点分析 · §3.3.2](./节点分析.md#332-ai_analyzer--需求分析器第一个-ai)

### 2.3 第三阶段：AI 规划器（`ai_planner`）

根据分析器生成的需求 JSON，交付给 **qwen3-turbo（或 GPT-5.2 / deepseek V3.2 等推理类模型）** 深入分析应该搭建什么样的工作流。

此过程中：
- **启用懒加载模式**展示"分析中..."
- 部分输出结果以 **JSON + 标签对** 形式呈现，便于提示用户进度
- 最终输出完整的 `{ nodes: [], edges: [] }` 结构，可直接用于 React Flow 渲染

> 📌 节点规划的具体可用节点池，详见 [节点分析 · §3.1 节点总览矩阵](./节点分析.md#31-节点总览矩阵)

### 2.4 第四阶段：逐节点执行与流式输出

工作流执行引擎按 edges 顺序遍历节点，逐节点调用 AI，通过 SSE 逐 token 流式推送到前端画布。

```
引擎执行循环：
  for each node in topological_sort(edges):
    1. 注入暗线全局上下文到 System Prompt
    2. 调用对应模型（根据 node.model_route）
    3. SSE 流式推送每个 token 到前端
    4. 前端实时更新 node.data.output + 状态变更
    5. 节点完成后，输出传递给下游节点
```

## 三、知识库 RAG 架构

> **设计定位：** 当数据获取模式选择"知识库"时，工作流禁止所有搜索环节，增加知识库检索节点。  
> **本质：** 一个**智能摘要驱动的分层 RAG 系统**，把大模型从"全文阅读器"变成了"摘要理解 + 精准定位"的工具。

### 3.1 核心思想：三层漏斗式处理

基本原则：**"不要让大模型每次都读全文"**。  
构建了一个"摘要 – 向量 – 原文"的三层漏斗：

| 层级 | 功能 | 覆盖度 | Token 消耗 |
|------|------|--------|-----------|
| **摘要层** | 文档上传后 AI 自动生成核心观点、目录架构和深度摘要（10万字→1万字） | 80% 的日常问答 | 极低 |
| **向量层** | 文档分块转为向量，语义检索找到最相关的原文片段 | 15% 需精准定位的问题 | 中等 |
| **原文层** | 极少数需精确引用或深度分析时，读取完整原文 | 5% 极端需求 | 高 |

### 3.2 文档预处理管道（用户上传后自动触发）

#### 文档解析
- 提取纯文本，按段落、标题等逻辑切分成块（chunk），每块约 500-800 tokens
- 保留上下文重叠

#### AI 总结
对每个文档调用一次大模型，生成：
- **摘要**：500-1000 字的精炼概括
- **核心观点**：用列表形式列出关键结论或数据
- **目录架构**：层级目录（JSON 格式），每个章节指向对应的原文位置（如页码、段落 ID）

#### 向量化存储
- 将每个文档的 **摘要** 和 **分块后的原文** 分别向量化，存入向量数据库（Supabase pgvector）
- 摘要、目录、核心观点等结构化数据存入关系数据库
- 原始文件本身存入对象存储（如硬盘、云存储）

### 3.3 问答时的三级检索逻辑

```
用户提问
    │
    ▼
┌──────────────────────────────┐
│ 第一步：检索摘要层              │
│  · 向量检索摘要库               │
│  · 找到最相关的文档摘要           │
│  · 摘要 + 问题 → LLM 尝试回答   │
│  · 能回答？→ 直接返回 ✅         │
│  · 不能？ → 进入第二步           │
└──────────────────────────────┘
    │
    ▼
┌──────────────────────────────┐
│ 第二步：深入向量层              │
│  · 向量检索原文分块库            │
│  · 找到最相似的 3-5 个原文片段    │
│  · 片段 + 摘要 + 问题 → LLM    │
│  · 生成精确答案 ✅              │
└──────────────────────────────┘
    │
    ▼
┌──────────────────────────────┐
│ 第三步：定位原文（极少数情况）    │
│  · 根据目录架构定位具体位置       │
│  · 返回完整原文段落 ✅           │
└──────────────────────────────┘
```

### 3.4 联网搜索的对称处理

- **多源并行搜索**：同时搜索权威源（学术、政府、新闻）和论坛源（Reddit、知乎、技术社区），每个结果附带来源标签
- **内容总结**：抓取网页内容，由 AI 生成综合摘要，附上原始链接
- **两级问答**：一般问题先用搜索结果摘要回答；需要深入原文时才展开具体网页内容

### 3.5 缓存机制

| 缓存类型 | 存储技术 | 用途 |
|---------|---------|------|
| **Token 缓存** | Redis 或数据库 | 高频查询的答案和上下文缓存，避免重复计算 |
| **向量缓存** | Redis 或内存 | 热门文档的向量检索结果缓存，减少 pgvector 查询压力 |

### 3.6 数据存储分层

| 层级 | 存储内容 | 存储技术 | 用途 |
|------|----------|----------|------|
| **文档原文** | 用户上传的原始文件（PDF、Word等） | 对象存储（本地硬盘/云OSS） | 极少访问，用于精确溯源 |
| **摘要层** | AI 生成的摘要、核心观点、目录架构 | 关系数据库（Supabase） | 快速回答常见问题，结构化查询 |
| **向量层** | 摘要的向量、原文分块的向量 | 向量数据库（pgvector） | 语义检索，定位相关片段 |
| **对话记录** | 用户提问、AI 回答、引用来源 | 关系数据库 | 历史追溯，个性化体验 |
| **缓存层** | 高频查询的答案和上下文 | KV 存储（Redis） | 降低重复计算，提升响应速度 |

### 3.7 知识库存储限制

| 限制项 | 规则 |
|--------|------|
| 每个知识库容量上限 | 100 MB |
| 用户知识库数量 | 无上限 |
| 总存储空间上限 | 1 GB（免费版） |
| 超出后处理 | 需付费升级，价格 ¥1/GB/月 |
| 支持格式 | Word、PDF、Markdown、TXT、图片 |

### 3.8 用户体验与成本优势

- **对用户**：文档上传后自动提炼，问答响应快，答案有据可查；联网搜索覆盖全面，来源清晰。
- **对开发者**：通过摘要层过滤掉 80% 的无效 token 消耗，成本大幅下降；缓存进一步优化；且整体架构易于扩展和维护。
- **与会员体系结合**：可将原文精确定位、联网搜索深度结果等作为付费功能，实现商业模式闭环。

## 四、核心创举：明暗线双轨架构

> **痛点分析：** 传统线性工作流（A → B → C）处理长文本时，大模型会"失忆"且"精神分裂"。负责写第一段的节点和负责写最后一段的节点，没有全局统筹，拼接出来的文章风格不统一，逻辑割裂。

### 4.1 暗线 · 全局上下文管理器 (Implicit Context)

| 维度 | 说明 |
|------|------|
| **本质** | 不是一个执行动作的节点，而是一个驻留在内存或数据库中的 **JSON 状态机** |
| **生命周期** | 从工作流开始生成，直到整个任务结束，一直存在且可以被动态更新 |

**暗线数据示例**（撰写一份社会实践调研报告场景）：

```json
{
  "global_theme": "XXX村产业调研",
  "language_style": "严谨的学术报告，多用客观数据",
  "core_outline": ["背景", "现状", "问题", "对策"],
  "target_audience": "高校评委"
}
```

### 4.2 明线 · 局部任务执行者 (Explicit Task)

| 维度 | 说明 |
|------|------|
| **本质** | 真正干活的单一节点，如"撰写【现状】部分" |
| **运行机制** | 当明线节点启动时，系统在后台自动把暗线数据作为 System Prompt 强行注入 |
| **最终效果** | 不管明线拆分成多少细碎任务，每一个 AI 节点都在潜意识里遵循同一个规则 |

### 4.3 明暗线协同的类比

> 这就像一个有着统一 **"总编（暗线）"** 审稿的 **"记者团队（明线节点）"**。

```
                    ┌── 暗线（总编）──────────────────────────────────┐
                    │  全局主题 · 语言风格 · 核心大纲 · 受众定位       │
                    └────┬──────────────┬──────────────┬──────────────┘
                         │              │              │
                    ╔════▼════╗    ╔════▼════╗    ╔════▼════╗
                    ║ 明线 A  ║    ║ 明线 B  ║    ║ 明线 C  ║
                    ║ 写前言  ║    ║ 写正文  ║    ║ 写结论  ║
                    ╚════╤════╝    ╚════╤════╝    ╚════╤════╝
                         │              │              │
                    └────┴──────────────┴──────────────┘
                         │
                    [merge_polish: 润色合并节点]
                         │
                    完整一致的长文输出
```

> 📌 暗线注入的技术实现细节和代码示例，详见 [节点分析 · §4.2 暗线注入机制](./节点分析.md#42-暗线注入机制)

## 五、动态打断与人机协同机制

> **设计理念：** 绝大多数 AI 工作流一旦运行就像脱缰野马，用户只能干等几分钟看盲盒结果。我们的设计将控制权交还给用户。

### 5.1 技术挑战

传统 HTTP 请求是同步的，一旦发出去很难优雅中断。

### 5.2 深层解决方案：异步任务队列

| 组件 | 说明 |
|------|------|
| **任务队列** | Task Queue（Redis 或 Supabase 配合 Edge Functions） |
| **状态字段** | 每个工作流在数据库中有实时状态：`running` / `paused` / `failed` |
| **非阻塞执行** | 引擎后端绝不能用死循环跑节点，必须用异步任务队列 |

### 5.3 "踩刹车"与"动手术"三步曲

#### Step 1：暂停 (Pause)

用户点击停止 → 数据库状态秒变 `paused` → 当前正在跑的节点跑完后，系统立刻掐断后续流程，进入"挂起"状态。

#### Step 2：人工覆写 (Manual Override)

场景：AI 生成的大纲严重跑题。

用户在界面上直接修改大纲文本框 → 前端向后端发送更新指令 → 数据库里这个节点的 Output 数据被**替换为用户手写版本**。

#### Step 3：满血复活 (Resume)

用户点击继续 → 引擎读取状态 → 发现到了下一步 → 带着用户刚刚修改过的、**绝对正确的**数据继续往下执行。

> 🎯 **核心理念**：这叫 **"人在回路（Human-in-the-Loop）"** —— AI 只是苦力，人才是真正的驾驶员。

## 六、拆解、聚合与格式化输出

> **痛点分析：** 大语言模型有"单次输出字数限制（Token Limit）"。10000 字的详细报告，哪怕最强模型也会中途截断或偷懒。

### 6.1 分拆（Map 阶段）：并行任务切片

系统根据**暗线**中的大纲，自动生成多个独立的生成任务：

```
例如：
  节点 A → 负责写"前言"
  节点 B → 负责写"正文"
  节点 C → 负责写"结论"

  ↳ 这些节点可以互不干扰地同时向大模型发起请求，极大缩短整体等待时间
```

### 6.2 聚合（Reduce 阶段）：灵魂缝合

这是流程中最关键的**隐藏节点**（即 [节点分析 · `merge_polish`](./节点分析.md#338-merge_polish--润色合并p1-阶段)）：

- 把 A、B、C 的文本直接物理拼接是没法看的 —— 会有生硬的过渡
- **"润色合并节点"**的任务：不产生新观点，而是梳理上下文衔接，消除重复话术（比如每个节点可能都自己写了总结词），使其变成一篇连贯的长文

### 6.3 实体化导出（Export）

对于学习赋能场景，最终交付物必须能直接拿去打印或提交。

后端利用 Python 强大的生态：

| 输出格式 | 技术方案 | 说明 |
|---------|---------|------|
| **PDF** | `reportlab` 或 `pdfkit` | 排版精美的 PDF |
| **DOCX** | `python-docx` | 兼容 Word 格式 |
| **Markdown** | 原始输出 | 在前端直接渲染 |

接到合并后的 Markdown 文本 → 一键转换 → 前端抛出下载链接。

## 七、可观测性：透明化运行看板

> **设计目标：** 建立用户信任，区分"黑盒工具"与"专业系统"。不仅给程序员看 Log，更要给普通用户看懂 AI 的思考过程。

### 7.1 呼吸感 UI 的底层逻辑 (WebSocket)

前端必须和后端建立 **WebSocket 长连接**。后端每执行完一步操作，就推一条消息：

```
后端推送：Node_3 状态变更为 running
  → 前端：第 3 个节点立刻亮起呼吸灯

后端推送：Node_3 状态变更为 success
  → 前端：呼吸灯停止，打上绿色对勾 ✅
```

### 7.2 数据"脱水"与标签化渲染

#### 原始 JSON（后端传来的数据）

```json
{
  "search_keywords": ["生物医药", "创新创业"],
  "limit": 10
}
```

#### UI 转化（前端组件识别到"搜索节点"数据）

```
🔍 正在搜索： 生物医药 | 创新创业
⚙️ 深度限制： 10条
```

- 直接屏蔽大括号，以友好的标签形式渲染
- 不同节点类型对应不同的渲染模板

### 7.3 Markdown 实时预览

如果节点产出的是文章片段，右侧边栏直接激活 **Markdown 渲染引擎**：

- 像打字机一样把带有标题、加粗、列表的排版内容**实时展示**
- 用户一边看着侧边栏的文字流，一边随时准备按下"暂停"键进行人工干预
