## 3.2 仍建议做真实页面手工验收

虽然逻辑测试已经补齐，但当前前端 Vitest 环境是偏 node 的纯逻辑验证，不是完整浏览器交互验收。因此以下场景仍建议人工走一遍：

### 场景 A：改配置后立即执行

目标：
- 确认用户修改节点配置后，不需要等待云同步完成，也能直接命中后端最新图结构。

重点观察：
- 执行结果是否体现最新 `node.data.config`
- 后端是否按 POST body 的图运行

### 场景 B：配置弹窗与执行面板互斥

目标：
- 打开某个节点配置后立刻点击运行，确认配置弹窗关闭，只保留执行面板。

重点观察：
- 是否仍存在双面板视觉冲突
- 是否有焦点残留或外层点击失效

### 场景 C：分支工作流的线路筛选

目标：
- 在包含 `logic_switch` 和汇聚节点的工作流中，确认 `全部 / 线路 N` 的筛选逻辑正确。

重点观察：
- 单链工作流是否不显示 Tab
- 多链工作流是否显示 Tab
- 汇聚节点是否在多条线路中都能出现

### 场景 D：右下角节点的配置弹窗翻转

目标：
- 节点靠近 viewport 右下边界时，确认弹窗自动翻转并仍在屏幕内。

重点观察：
- 是否超出右边界
- 是否超出下边界
- 是否因为缩放或滚动出现位置偏差

## 4. 哪些内容是有意不做，不算漏项

以下内容是经过判断后刻意不做的，不应在后续复盘时被认定为遗漏：

## 4.1 不做 POST 执行自动重试

原因：
- 执行接口有副作用
- 自动重试可能导致重复执行

当前策略：
- 流异常中断时直接把 session 标记为 `error`
- 由用户手动重试

这是设计选择，不是半成品。

## 4.2 不把 `ExecutionTraceDrawer` 改成隐藏态常驻 DOM

原因：
- 当前抽屉逻辑简单，条件渲染更干净
- 没必要为了一个固定面板额外引入隐藏态状态机

当前策略：
- 有 session 才渲染
- 初次挂载时做轻量 enter animation

这也是设计选择，不是遗漏。

## 4.3 不强制把 `NodeConfigDrawer.tsx` 改名为 `NodeConfigPopover.tsx`

原因：
- 当前文件语义虽然已转为 Popover，但改名不带来实际收益
- 改名会扩大改动面和引用风险

当前策略：
- 保留旧文件名
- 在文档里明确说明其行为已是 popover

这属于命名保守策略，不影响功能。

## 4.4 不在本轮补浏览器级组件测试栈

原因：
- 当前仓库的 Vitest 主要跑 node 环境纯逻辑测试
- 若临时引入 jsdom / testing-library，会扩大依赖和测试基建范围

当前策略：
- 先把定位、配置 patch、事件更新、trace filtering 这些核心逻辑抽成 helper 并补纯逻辑测试
- 真实交互由手工验收兜底

这属于“控制改动边界”，不是测试偷工减料。
