# 工作流记忆系统（Workflow Memory）— 深度分析

> 创建日期：2026-03-30
> 关联特性：AI 对话记忆持久化、工作流运行日志云端同步、m/[id] 动态路由

---

## 一、问题域分析

### 1.1 当前痛点

**AI 对话记忆无法长期保存**

- 当前 AI 对话存储在 `ss_ai_conversations` + `ss_ai_messages` 表中，消息本身可持久化
- 但对话上下文（context window）受模型限制，长对话会丢失早期信息
- 用户关闭页面后重新打开，虽然历史消息可加载，但 AI 无法"记住"之前的推理过程和决策
- 工作流执行过程中每个节点产生的中间结果（`NodeExecutionTrace`）仅存在于前端 Zustand store 的 `executionSession` 中，页面刷新即丢失

**工作流运行日志不可回溯**

- `ss_workflow_runs` 表记录了运行的基本信息（status、tokens_used、started_at、completed_at）
- `output` JSONB 字段仅保存 `workflow_status`（最终状态）和 `saved_results`（write_db 节点写入的数据）
- 每个节点的详细执行过程（输入快照、流式输出、执行时间、错误信息）没有持久化到数据库
- 用户无法在事后查看"第 3 个节点当时输入了什么、输出了什么、花了多长时间"

**缺少独立的运行日志分享机制**

- 当前分享机制（`/s/[id]`）是分享整个工作流的画布结构，需要工作流设为 `is_public`
- 没有办法分享某一次具体的运行结果（"看看我这次跑出来的分析报告"）
- 运行日志的分享需求与工作流结构的分享需求是不同的：
  - 工作流分享 = 分享"怎么搭建的"（结构 + 节点配置）
  - 运行日志分享 = 分享"跑出来了什么"（每个节点的输入输出结果）

### 1.2 用户场景

| 场景 | 描述 | 当前支持 |
|------|------|---------|
| 回顾学习记录 | 用户想查看上周某次工作流的分析结果 | ❌ 前端 session 已丢失 |
| 对比多次运行 | 同一工作流不同输入的结果对比 | ❌ 只有最新一次的 nodes_json |
| 分享运行成果 | 把某次运行的闪卡/思维导图分享给同学 | ❌ 只能分享工作流结构 |
| AI 记忆延续 | AI 对话中引用之前运行的结论 | ❌ 上下文窗口限制 |
| 知识库沉淀 | 将多次运行的精华内容归档到知识库 | ⚠️ 仅 write_db 节点可部分实现 |

## 二、现有系统架构分析

### 2.1 数据层现状

```
ss_workflows (工作流定义)
  ├── nodes_json    → 节点结构 + 最新一次执行的 output（会被覆盖）
  ├── edges_json    → 连线结构
  └── is_public     → 控制 /s/[id] 是否可访问

ss_workflow_runs (运行记录)
  ├── input         → 触发输入文本
  ├── output        → { workflow_status, saved_results }（仅最终状态）
  ├── status        → running / completed / failed
  ├── tokens_used   → 总 token 消耗
  ├── current_step  → 当前步骤（执行中使用）
  └── total_steps   → 总步骤数

ss_ai_conversations (AI 对话)
  └── ss_ai_messages (对话消息)
      ├── content       → 消息文本
      ├── actions_json  → AI 执行的操作
      └── canvas_snapshot → 画布快照（可选）
```

### 2.2 前端执行会话结构

```typescript
// 当前 WorkflowExecutionSession（仅存在于内存）
{
  sessionId: string;          // 前端生成的 UUID
  workflowId: string;
  workflowName: string;
  startedAt: number;          // performance.now()
  finishedAt?: number;
  totalDurationMs?: number;
  overallStatus: 'running' | 'completed' | 'error';
  traces: NodeExecutionTrace[];  // ← 核心：每个节点的详细执行记录
  completedCount: number;
  totalCount: number;
  chains?: WorkflowChain[];
}

// NodeExecutionTrace — 节点级执行详情
{
  nodeId, nodeType, nodeName, category, status,
  executionOrder, startedAt, finishedAt, durationMs,
  isParallel, parallelGroupId,
  inputSummary, rawInputSnapshot,   // 输入
  streamingOutput, finalOutput,     // 输出
  outputFormat, errorMessage,       // 格式 + 错误
  modelRoute, chainIds,             // 模型 + 链路
}
```

### 2.3 /s/[id] 路由模式分析

```
frontend/src/app/s/[id]/
  ├── layout.tsx              → 独立布局（不走 dashboard layout）
  │   ├── SessionRefresher    → Supabase auth 会话维持
  │   └── 顶部浮动导航栏      → "← StudySolo" + "公开预览"
  ├── page.tsx                → Server Component，调用 fetchPublicWorkflowForServer
  └── PublicWorkflowView.tsx  → Client Component，渲染只读画布 + 社交互动
```

关键设计决策：
- Server Component 负责数据获取（带 cookie 鉴权）
- 独立 layout 不依赖 dashboard 的 AuthProvider
- `SessionRefresher` 确保匿名/已登录用户都能正常使用
- 公开访问由 `is_public` 字段控制，后端 RLS 策略放行

### 2.4 当前分享机制

- 工作流分享链接格式：`{origin}/s/{workflowId}`
- 生成位置：`CollaborationPopover.tsx` 中的"公开分享链接"区域
- 前提条件：工作流必须设为 `is_public = true`
- 后端 API：`GET /api/workflow/{id}/public`（无需鉴权，但可选传 token 获取个性化数据）

## 三、m/[id] 路由与 s/[id] 的核心区别

| 维度 | s/[id]（工作流分享） | m/[id]（运行记忆） |
|------|---------------------|-------------------|
| 展示内容 | 工作流画布结构（节点 + 连线） | 所有节点的执行结果（输入/输出纸条） |
| 数据来源 | `ss_workflows.nodes_json` + `edges_json` | `ss_workflow_run_traces`（新表） |
| 访问控制 | `is_public` 字段控制 | 独立的 `is_shared` 开关，默认私有 |
| 交互能力 | 点赞、收藏、Fork | 仅查看（可选：评论） |
| 时间维度 | 展示最新版本的工作流 | 展示某一次具体运行的快照 |
| 所有者操作 | 编辑工作流 | 标注/备注、分享开关 |

## 四、数据模型需求分析

### 4.1 需要新增的数据

**节点级执行记录（核心缺失）**

当前 `ss_workflow_runs.output` 只存最终状态，缺少每个节点的：
- 输入快照（input_snapshot）
- 完整输出（final_output）
- 流式输出格式（output_format）
- 执行时间（duration_ms）
- 使用的模型（model_route）
- 执行顺序和并行信息
- 错误信息

**运行级分享控制**

当前分享粒度是工作流级别（`ss_workflows.is_public`），需要运行级别的：
- `is_shared` 布尔字段
- `share_token`（可选：用于生成不可猜测的分享链接）
- `shared_at` 时间戳

### 4.2 新表设计方向

```sql
-- 方案 A：新增 ss_workflow_run_traces 表（推荐）
-- 每个节点一行，与 ss_workflow_runs 关联
CREATE TABLE ss_workflow_run_traces (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  run_id UUID NOT NULL REFERENCES ss_workflow_runs(id) ON DELETE CASCADE,
  node_id TEXT NOT NULL,           -- 对应 nodes_json 中的节点 ID
  node_type TEXT NOT NULL,
  node_name TEXT NOT NULL,
  category TEXT,
  execution_order INTEGER,
  status TEXT NOT NULL,            -- done / error / skipped
  input_snapshot TEXT,             -- 节点输入快照
  final_output TEXT,               -- 节点最终输出
  output_format TEXT,              -- markdown / json / flashcard 等
  duration_ms INTEGER,
  model_route TEXT,                -- 使用的模型 SKU ID
  is_parallel BOOLEAN DEFAULT FALSE,
  parallel_group_id TEXT,
  error_message TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- 方案 B：扩展 ss_workflow_runs.output JSONB
-- 在 output 字段中增加 traces 数组
-- 优点：不需要新表；缺点：JSONB 查询性能差，单行数据可能很大
```

### 4.3 分享控制扩展

```sql
-- 在 ss_workflow_runs 表上新增字段
ALTER TABLE ss_workflow_runs ADD COLUMN is_shared BOOLEAN DEFAULT FALSE;
ALTER TABLE ss_workflow_runs ADD COLUMN share_token TEXT UNIQUE;
ALTER TABLE ss_workflow_runs ADD COLUMN shared_at TIMESTAMPTZ;

-- 新增 RLS 策略：允许通过 share_token 公开访问
CREATE POLICY ss_workflow_runs_select_shared
  ON ss_workflow_runs FOR SELECT
  USING (is_shared = TRUE);
```

## 五、与知识库的关系

### 5.1 知识库现状

- `ss_kb_documents`：用户上传的文档（PDF 等）
- `ss_kb_document_chunks`：文档分块 + embedding
- 知识库面板（`KnowledgeBasePanel`）：展示工作流中有输出的节点（flashcard/quiz/mind_map/export_file）
- 知识库的"所有记录"按钮：应该能查看所有历史运行的节点输出

### 5.2 记忆系统与知识库的交叉

```
知识库（Knowledge Base）
  ├── 文档知识：上传的 PDF/文档 → 分块 → embedding → RAG 检索
  └── 运行知识：工作流运行产生的结果 → 节点输出 → 可检索/可浏览
                                          ↑
                                    这就是 Memory 系统要补的

Memory 系统（m/[id]）
  ├── 完整运行日志：每个节点的输入/输出/状态/时间
  ├── 分享能力：独立的分享开关
  └── 浏览入口：
      ├── 工作流右上角"运行日志"按钮 → 该工作流的所有运行
      └── 知识库"所有记录"按钮 → 所有工作流的所有运行
```

## 六、云端同步工作流分析

### 6.1 当前同步状态

| 数据 | 存储位置 | 云端同步 |
|------|---------|---------|
| 工作流结构 | Supabase `ss_workflows` | ✅ 实时同步 |
| 运行基本信息 | Supabase `ss_workflow_runs` | ✅ 后端直接写入 |
| 节点执行详情 | 前端 Zustand store | ❌ 仅内存 |
| AI 对话历史 | Supabase `ss_ai_messages` | ✅ 实时同步 |
| AI 对话上下文 | 模型 context window | ❌ 窗口限制 |

### 6.2 需要补充的同步

**每次完整运行的节点级数据 → 云端**

- 时机：工作流执行完成后（`workflow_done` 事件）
- 数据：`WorkflowExecutionSession.traces` 数组
- 方式：
  - 方案 A（后端写入）：在 `_finalize_run` 中，将执行过程中收集的节点数据批量写入 `ss_workflow_run_traces`
  - 方案 B（前端上报）：前端在 `finalizeExecutionSession` 后，调用新 API 上报 traces
  - 推荐方案 A：后端已经有每个节点的 SSE 事件数据，可以在执行过程中累积

### 6.3 后端执行引擎改造点

当前 `execute_workflow` 生成器产出 SSE 事件字符串，但不保留节点级数据。需要：

1. 在 `event_generator()` 中累积每个节点的执行数据
2. 解析 `node_input`、`node_token`、`node_done`、`node_status` 事件
3. 在 `_finalize_run` 时批量写入 `ss_workflow_run_traces`

```python
# 伪代码示意
node_traces = {}  # node_id -> trace_data

async for event in execute_workflow(...):
    yield event
    if evt := _parse_sse_data(event):
        if event.startswith("event: node_input"):
            node_traces[evt["node_id"]] = {"input_snapshot": evt["input_snapshot"], ...}
        elif event.startswith("event: node_done"):
            node_traces[evt["node_id"]]["final_output"] = evt["full_output"]
        # ... 其他事件

# 执行完成后批量写入
await _save_traces(service_db, run_id, node_traces)
```

## 七、前端入口分析

### 7.1 工作流右上角"运行日志"入口

当前工作流画布右上角区域（`WorkflowCanvas.tsx` 的工具栏）：
- 已有：分享/协作按钮（`CollaborationPopover`）
- 新增：运行日志按钮 → 点击后展示该工作流的所有运行记录列表
- 每条记录可点击跳转到 `/m/[runId]`

### 7.2 知识库"所有记录"入口

当前知识库面板（`KnowledgeBasePanel`）：
- 已有：按类型筛选的节点输出列表
- 新增："所有记录"按钮/Tab → 展示所有工作流的所有运行记录
- 按时间倒序排列，支持搜索和筛选

### 7.3 m/[id] 页面本身

- 展示某一次运行的完整记录
- 核心区别于 s/[id]：不是展示画布，而是展示每个节点的"纸条"（输入/输出卡片）
- 布局参考执行面板（`RightPanelContent` + `TraceStepItem`），但以全页面形式展示
- 顶部信息：工作流名称、运行时间、总耗时、状态、token 消耗
- 主体：按执行顺序排列的节点卡片，每张卡片显示：
  - 节点名称 + 类型图标
  - 输入摘要
  - 完整输出（支持 Markdown 渲染）
  - 执行时间 + 使用模型
- 分享开关：仅所有者可见，点击后生成分享链接
