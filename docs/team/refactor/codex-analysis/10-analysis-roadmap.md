# 正式分析与后续重构路线图

## 1. 当前阶段定义

本文件不是直接实施路线，而是从分析走向实施的顺序控制表。

目的：

- 防止在事实未冻结前直接重构。
- 防止在术语未统一前直接分工。
- 防止在接口未冻结前让队友并行开发。

## 2. 第一阶段：事实冻结

### 输出物

- 当前仓库结构图
- 当前运行链路图
- 当前节点、插件、社区节点、子后端概念对照表
- 当前文档与代码漂移清单

### 完成标志

- 团队能对“现在是什么”达成一致。

## 3. 第二阶段：术语与边界冻结

### 输出物

- 节点定义
- 插件定义
- 社区节点定义
- 子后端定义
- Wiki 边界定义

### 完成标志

- 团队不再用同一个词指代不同东西。

## 4. 第三阶段：接口冻结

### 重点冻结对象

- 节点 manifest 基础字段
- 插件模块元数据
- 子后端 gateway 契约
- Wiki 文档发布清单

### 完成标志

- 可为不同队友划分独立任务而不互相踩接口。

## 5. 第四阶段：模块拆分设计

### 优先顺序

1. 节点系统单一事实源化
2. 插件模块标准化
3. 子后端 gateway 设计
4. Wiki 初始化与发布链路

### 完成标志

- 已得到清晰的模块树和 ownership 方案。

## 6. 第五阶段：实施路线

### 建议的先后次序

1. 先做节点系统单一事实源化。
2. 再做插件模块标准化。
3. 再做子后端接入层。
4. 同时启动 Wiki 子项目，但只承接稳定文档。
5. 最后再考虑根级目录重命名和更重的 monorepo 工具化。

## 7. 推荐的队友任务拆分

### 小陈：Wiki

- 初始化 Wiki 子项目
- 承接稳定文档发布
- 建立主仓 -> Wiki 的发布清单

### 小李：子 Agent / 子后端

- 设计并实现一个最小子后端样板
- 对齐 OpenAI 兼容请求/响应/流式/错误结构

### 羽升

- 处理节点系统单一事实源化
- 处理插件模块标准化
- 维护主后端与主前端边界

## 8. 当前不建议立即执行的事项

- 不建议现在就重命名整个仓库目录树。
- 不建议现在就把所有现有节点改成插件或子后端。
- 不建议现在就让 Wiki 承担设计文档主来源。

## 9. 本文件结论

最正确的顺序不是“立刻大重构”，而是：

> 先冻结事实与边界，再冻结接口，再拆模块，再分团队，再实施。
