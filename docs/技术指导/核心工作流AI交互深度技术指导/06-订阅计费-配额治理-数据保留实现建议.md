# 订阅计费、配额治理、数据保留实现建议

## 1. 与你现有会员规划的技术拆分

建议把“价格方案”和“资源可用性”分离：

- `plans`：套餐定义（免费/Pro/Pro+/Ultra）
- `subscriptions`：用户当前订阅状态
- `entitlements`：当前可用配额（存储、并发、工作流数、日执行数）
- `add_ons`：按量加购项
- `usage_ledger`：每次扣减与恢复的流水账

## 2. 配额扣减原则

1. 并发配额
- 在“进入 running”前原子占位。
- 在 `done/error/paused` 释放配额。

2. 存储配额
- 以对象存储实际大小为准。
- 采用异步回写，账本记录变更前后值。

3. 日执行上限
- 按用户时区自然日结算，避免 UTC 误差争议。

## 3. 数据保留（对应你的 0/7/14/22 天规则）

- 设计状态：`active -> grace -> pending_delete -> deleted`
- 删除采用“两阶段”：
  1. 逻辑删除（不可见）
  2. 延迟物理删除（可恢复窗口）

## 4. 支付与计费文档建议

- 国际订阅与用量计费：Stripe Billing 文档。
- 跨境收单场景：Alipay+ 文档。
- 微信支付场景：微信支付 API v3 文档。

> 说明：支付链路强依赖公司主体、结算地区与合规策略，技术实现应在法务/财务确认后锁定。

## 5. 风险控制

- 所有计费事件必须幂等（外部 webhook 重放可安全处理）。
- 金额与配额变更分离记录，防止“扣费成功但配额未生效”。
- 提供管理员补偿接口（补发额度/回滚扣减）。
