# 工作流引擎：状态机、暂停恢复、人工覆写

## 1. 对应你当前 core 规划的执行模型

从 `trigger_input -> ai_analyzer -> ai_planner -> 执行节点` 的链路看，推荐采用：

- DAG 编排：`nodes + edges` 入库，执行前做拓扑排序。
- 节点状态机：`pending -> running -> done | error | paused`。
- 运行检查点：每个节点执行前/后都检查 `workflow.status`。

## 2. 最适配实现（P0 与 P1）

### P0（先落地）

- FastAPI 请求触发 + asyncio worker 协作式执行。
- 关键表：
  - `workflows`（整体状态）
  - `workflow_nodes`（节点定义）
  - `workflow_runs`（一次运行实例）
  - `node_runs`（每个节点的执行与输出）
  - `node_override_logs`（人工覆写日志）

### P1（增强）

- 引入 Celery + Redis：
  - 重任务（解析、向量化、导出）与执行任务解耦。
  - 更稳的重试、退避、死信与多 worker 扩缩。

## 3. 暂停/恢复与人工覆写的正确边界

1. 暂停
- 不是“杀进程”，而是“停止下游调度”。
- 当前 running 节点可设两种策略：
  - 软暂停：当前节点自然完成后暂停。
  - 硬暂停：在模型流式循环中检查取消标志并中断。

2. 人工覆写
- 覆写的是 `node_run.output` 的新版本，不覆盖历史版本。
- 恢复时总是读取该节点“已生效版本”继续下游。

3. 并发与幂等
- 每个 `workflow_run` 需要幂等锁（防止重复点击运行）。
- 数据库锁建议：事务级锁或唯一约束 + 状态条件更新。

## 4. 关键实现细节

- 拓扑排序：Python `graphlib.TopologicalSorter`。
- 取消语义：`asyncio.CancelledError` 必须可传播，避免“僵尸任务”。
- 失败重试：仅对可恢复错误重试（超时、429、瞬时网络）。

## 5. 不建议做法

- 仅在前端做暂停标记，不改后端状态。
- 覆写直接覆盖原始输出（丢失审计链）。
- 在长循环中没有取消检查点。
