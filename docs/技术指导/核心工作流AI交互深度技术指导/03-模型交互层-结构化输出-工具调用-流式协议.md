# 模型交互层：结构化输出、工具调用、流式协议

## 1. 与 `ai_analyzer` / `ai_planner` 的强匹配建议

你的两个关键节点都要求稳定 JSON 输出，因此推荐：

- 首选：Structured Outputs（strict JSON Schema）。
- 兜底：Pydantic 校验 + 自动修复重试（最多 2~3 次）。

OpenAI 文档已明确：Structured Outputs 相对 JSON mode 更可靠，且建议优先使用。

## 2. 为什么要“双轨”而不是只用 strict

原因：供应商兼容并非能力完全一致。

- OpenAI 原生：Structured Outputs 能力完整。
- 火山方舟某些接入形态（模型单元）官方限制中提到：不支持 `response_format` 的结构化输出场景。
- 结论：必须做能力探测与降级，不能把 strict 作为唯一依赖。

## 3. 工具调用策略（为后续节点扩展预留）

`ai_planner` 阶段可逐步引入 function tools：

- `search_docs`（知识库检索）
- `search_web`（联网搜索）
- `export_file`（导出任务投递）

建议：
- P0 先只做纯模型 JSON 输出。
- P1 再引入 tools，避免一开始调试面过大。

## 4. 流式协议建议

- 模型端流式：使用 SDK 流事件。
- 后端对前端：统一转成 SSE 事件包。

推荐事件类型：
- `node.started`
- `node.token`
- `node.usage`
- `node.done`
- `node.error`

## 5. 模型路由建议（结合你当前规划）

- `ai_analyzer`：偏短文本结构化提取，走成本优先模型。
- `ai_planner`：偏推理与编排，走推理质量优先模型。
- `merge_polish`：高 token 节点，必须启用上下文预算与截断策略。

## 6. 强制工程规范

- 每次模型调用都记录：`provider/model/input_tokens/output_tokens/latency/request_id`。
- 输出入库前必须通过 schema 校验。
- 任意失败都要保留原始响应与错误码用于排查。
