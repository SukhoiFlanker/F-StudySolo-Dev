## 3. 为什么是这个顺序

### 3.1 为什么先整规范（第一波）

- Gateway 开发涉及三人协作，规范不统一会导致各自理解不同的"规范"
- 规范整理本身就是 Task 5.4 的核心内容
- 整理过程会自然暴露问题优先级

### 3.2 为什么治理在 Gateway 之前（第二波）

- 裸 fetch 问题如果不修，Gateway 接入后前端调用 Agent 时可能继续绕过认证
- ESLint 规则和 CI 在 Gateway 开发期间就能开始保护代码质量
- CODEOWNERS 补全后，三人并行推进时 PR review 有明确责任人

### 3.3 为什么 Gateway 是第三波而不是第一波

- Gateway 是最大的新增代码量，需要在规范和治理基础稳定后再动手
- 但 Gateway 也是 Phase 5 的核心交付物，不能拖太久
- 当前已有 `code-review-agent` 作为真实接入对象，不需要等其他 Agent

### 3.4 为什么 Wiki 放最后

- Wiki 由小陈并行推进，不阻塞主线
- 当前 owner 只需要提供骨架和目录结构
- Wiki 内容填充是长期工作，不在 Phase 5 scope 内

## 4. 关键设计决策（已确认）

| 决策点 | 结论 | 理由 |
|--------|------|------|
| Agent 调用 AI 的方式 | 独立调用，各自管理 API Key | 已是事实（`code-review-agent` 的 `AGENT_UPSTREAM_*`） |
| Gateway 职责 | 纯路由转发 + 治理，不碰 AI Provider | Agent 独立调用模式下 Gateway 不需要代理 AI |
| Gateway vs LLM Router | 无职责重叠 | LLM Router 服务节点执行，Gateway 服务 Agent 调用 |
| `agents.yaml` hot-reload | 启动时加载一次 | 简单可靠，避免运行时状态不一致 |
| 熔断器 | `pybreaker` 进程内 | 单 worker 部署下够用 |
| SSE 透传 | 流式转发，不 buffer | Gateway 必须支持 SSE 透传 |
| Wiki 方案 | Next.js Route Group 嵌入主前端 | 独立子项目方案已废弃 |

## 5. 风险与禁区

### 5.1 风险点

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 规范整合时误删有效内容 | 后续开发缺少参考 | 整合前 git commit 快照 |
| Gateway SSE 透传实现复杂度 | 可能超出预估时间 | 先实现 non-stream，再补 stream |
| 三人并行时 merge 冲突 | 开发效率下降 | CODEOWNERS + 路径触发 CI |
| 前端裸 fetch 修复引入回归 | 功能异常 | 逐文件修复，每次修复后手动 smoke |

### 5.2 禁区

- 不回头深挖 `code-review-agent` 4B 细节
- 不补其他 Agent 骨架（小李的活）
- 不碰工作区无关脏文件
- 不重开 Phase 2/3 大重构
- 不重命名目录树或引入 monorepo 工具链
- 不把 Wiki 当设计源
