## 2. 整合方案

### 2.1 目标结构

```
docs/项目规范与框架流程/项目规范/
├── 00-规范索引.md              ← 新建：统一入口，列出所有规范及权威层级
├── 01-项目架构全景.md          ← 保留：唯一的架构总览（去掉 API 路由表重复）
├── 02-模块边界规范.md          ← 保留：吸收 frontend-engineering-spec 中的边界部分
├── 03-命名规范.md              ← 重命名自 naming.md
├── 04-API规范.md               ← 重命名自 api.md
├── 05-设计规范.md              ← 重命名自 design.md
├── 06-节点开发规范.md          ← 重命名自 03-插件开发规范.md，去掉 Agent 重复
├── 07-子后端Agent规范.md       ← 保留：去掉协议重复，改为引用 agent-architecture.md
├── 08-前端工程规范.md          ← 重命名自 frontend-engineering-spec.md，去掉与 02 的重叠
└── 09-AI调用与计费规范.md      ← 保留现有的统一规范
```

### 2.2 整合原则

1. **每个主题只有一个权威文档**——消除"同一件事在两个地方说"
2. **`final-plan/` 只保留执行计划和阶段状态**——不再承担"规范"职责
3. **`agent-architecture.md` 保留为协议冻结文档**——`07-子后端Agent规范.md` 引用它而不是复制它
4. **`project-context/SKILL.md` 只做索引和摘要**——不重复完整规范内容
5. **整合前先 git commit 快照**——防止误删有效内容

### 2.3 文档权威层级（整合后）

| 优先级 | 文档位置 | 说明 |
|--------|---------|------|
| **L0（最高）** | `docs/team/*.md` | 团队协作铁规（commit/PR/issue/角色/接口同步） |
| **L0** | `.github/CODEOWNERS` | 代码所有权 |
| **L0** | `docs/team/refactor/contracts/` | Phase 1 冻结契约（不可修改） |
| **L1** | `docs/项目规范与框架流程/项目规范/` | 技术规范主体（整合后的统一入口） |
| **L1** | `docs/team/refactor/final-plan/` | 重构执行计划（只记录阶段状态，不重复规范） |
| **L2** | `agents/README.md` | Agent 开发指南（引用 L1 规范） |
| **L2** | `.agent/skills/project-context/SKILL.md` | AI 编程速查（索引 + 摘要，不重复规范） |
| **L3（参考）** | `docs/team/refactor/claude-analysis/` | 历史分析，只读 |
| **L3（参考）** | `docs/team/refactor/codex-analysis/` | 历史分析，只读 |
