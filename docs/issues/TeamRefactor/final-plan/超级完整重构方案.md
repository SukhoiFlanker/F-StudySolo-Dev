## 第一部分：核心原则（Codex 骨架）

### 1.1 四大不可违背原则

#### 原则一：单一事实源（Single Source of Truth）

**定义：** 节点系统、插件系统、配置系统必须各自只有一个真实来源。

**具体要求：**
- 节点定义只存在于 `backend/app/nodes/` 或 `packages/shared/nodes/manifest.json`
- 禁止在多个目录或文件中重复定义同一节点
- 所有前端渲染器必须与后端节点一一对应

**违反后果：**
- 多人协作时出现"节点不一致"问题
- 运行时加载的节点与文档描述不符
- 维护成本成倍增加

---

#### 原则二：先冻结契约再并行开发（Freeze Contract Before Parallel Dev）

**定义：** 在团队成员并行开发之前，必须先冻结模块间的接口契约。

**四阶段执行路线（来自 Codex）：**

| 阶段 | 内容 | 完成标志 |
|------|------|---------|
| 阶段一 | 冻结事实 | 团队对"现在是什么"达成一致 |
| 阶段二 | 冻结术语与边界 | 团队不用同一个词指代不同东西 |
| 阶段三 | 冻结接口 | 可为不同队友划分独立任务而不互相踩接口 |
| 阶段四 | 实施路线 | 已得到清晰的模块树和 ownership 方案 |

---

#### 原则三：Wiki 作为发布源而非设计源

**定义：** Wiki 只承接已稳定的文档，新设计文档保留在主仓。

**单向发布关系：**
```
主仓 docs/  →  (发布)  →  Wiki
   (设计源)                 (发布源)
```

**禁止事项：**
- 禁止在 Wiki 上做设计讨论
- 禁止 Wiki 作为多人协作的设计文档主战场
- 设计文档始终在主仓 `docs/plan/` 下

---

#### 原则四：子后端四层兼容性

**定义：** 子后端的 OpenAI 兼容不只是字段兼容，而是平台级四层兼容。

| 层级 | 内容 | 最低要求 |
|------|------|---------|
| 请求兼容 | model/messages/温度/流式/工具 | 必须 |
| 响应兼容 | 非流/流式 chunk/错误格式 | 必须 |
| 运行时兼容 | 超时/重试/幂等/速率限制 | 必须 |
| 平台治理兼容 | API Key 传递/审计/用量统计/版本标识 | 必须 |

---

### 1.2 术语精确定义（Codex）

| 术语 | 定义 | 示例 |
|------|------|------|
| **节点 Node** | 工作流执行单元，由主引擎调度 | `QuizGenNode`, `InputNode` |
| **插件 Plugin** | 仓内独立模块，可暴露前端/后端/数据库/节点 | `AnkiExporterPlugin` |
| **社区节点 Community Node** | 用户发布的节点能力描述/模板，非直接运行代码 | marketplace 上的分享 |
| **子后端 Sub-backend** | 团队成员独立开发的服务，通过协议接入 | `CodeReviewAgent`, `DocGeneratorAgent` |

## 第四部分：子后端 Agent 架构

### 4.1 Agent Gateway 架构

```
frontend
   ↓
main backend
   ↓
Agent Gateway（新增层）
   ↓
┌───────────────┬───────────────┬───────────────┐
│ sub-backend A │ sub-backend B │ sub-backend C │
│ (CodeReview)  │ (DocGen)      │ (Custom)      │
└───────────────┴───────────────┴───────────────┘
```

**Gateway 职责：**
- 服务发现
- 统一认证
- 统一请求协议转换
- 统一流式事件映射
- 统一错误归一化
- 统一计量与审计

```
data: {"id":"chatcmpl-xxx","object":"chat.completion.chunk","choices":[{"index":0,"delta":{"content":"..."},"finish_reason":null}]}

```

### 6.2 依赖规则

**前端依赖方向：**
```

stores/         ←  components/ 和 hooks/ 可依赖
hooks/          ←  components/ 可依赖
features/       ←  不能相互依赖（除非通过 public API）
services/       ←  features/ 可依赖
types/          ←  所有模块可依赖（只读）
```

**后端依赖方向：**
```

core/           ←  api/, services/, engine/ 可依赖
services/       ←  api/ 可依赖
api/            ←  不能依赖 engine/（防止循环）
engine/         ←  nodes/ 可依赖
nodes/          ←  不能依赖 api/, services/
```
