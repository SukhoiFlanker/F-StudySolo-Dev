# 节点、插件与 SOP 重构策略

## 1. 当前最核心的判断

现有节点系统本身不是失败的，相反，它已经有一条很好的基础路线：

- 后端节点目录化。
- `BaseNode` 自动注册。
- `/api/nodes/manifest` 输出元数据。
- 前端有独立渲染器注册。

真正的问题在于：

> 节点体系已经开始平台化，但前端和协作文档还没有完全转向平台化思维。

## 2. 节点未来应坚持的原则

### 2.1 一个节点应当以“单目录可理解”为最小交付单元

未来新增节点时，最理想的工程体验应是：

- 主要后端逻辑在一个节点目录内可理解。
- Prompt、config schema、输出约束尽量跟节点本体靠近。
- 前端只保留必要的渲染与交互扩展。

这与用户提出的“最好只在一个文件夹创建新的节点”方向是一致的。

但这里要注意：

- “一个文件夹完成全部工作”更适合作为设计目标。
- 对前后端分离系统来说，不可能严格只改一个物理目录。
- 更准确的目标应是“一个节点 = 一个主目录 + 少量标准注册点”。

## 3. 现有 SOP 是否还要保留

### 结论

保留，但必须升级。

### 原因

现有 SOP 最大价值不在于文件名本身，而在于它已经建立了重要的分类思维：

- A 型：纯 LLM 节点
- B 型：外部工具节点
- C 型：插件

这套思路应该继续保留，因为它能防止团队成员一上来就把所有扩展都写成同一种东西。

### 需要升级的地方

1. 从“实现清单”升级为“模块边界清单”。
2. 明确节点、插件、社区节点、子后端的区别。
3. 让 SOP 直接面向多人协作，而不只是面向单人开发流程。
4. 将“manifest-first”纳入节点开发标准。

## 4. 建议的未来术语重定义

### 4.1 节点 Node

定义：

- 工作流中的执行单元。
- 由主引擎调度。
- 必须遵守统一的 Node Contract。

### 4.2 插件 Plugin

定义：

- 仓库内独立模块。
- 可以拥有自己的前端页面、后端 API、数据库表、节点接入点。
- 节点只是它的可选暴露方式之一。

### 4.3 社区节点 Community Node

定义：

- 由用户发布的节点能力描述。
- 当前更接近“可配置的社区节点模板/资产”，不是完整插件。

### 4.4 子后端 Sub-backend

定义：

- 由团队成员独立开发的后端模块或服务。
- 通过统一协议接入主平台。
- 可以承载特定 Agent 或插件能力。

## 5. 节点系统的目标架构

### 5.1 后端

后端继续作为运行时权威源：

- `node_type`
- `category`
- `description`
- `config_schema`
- `supports_upload`
- `supports_preview`
- 输出能力

### 5.2 前端

前端保留：

- 渲染器选择
- 交互增强
- 视觉分组

但不应再继续成为节点基础事实的主定义地。

### 5.3 商店

节点商店未来应分为至少三层：

- 内置节点
- 社区节点
- 插件暴露节点

不能再把“默认节点”和“共享节点”作为唯一二分法。

## 6. 插件的目标结构

未来每个插件都应是“仓内独立模块”，至少应包含：

- 前端功能入口
- 后端 API
- 自己的文档
- 是否暴露节点的声明
- 是否依赖子后端的声明

插件不等于：

- 一个 mock 卡片
- 一个单独 node.py
- 一个 README 想法

## 7. 推荐的 SOP 升级方向

### 新增一层：模块决策门

在 A/B/C 分类之前，先判断新增内容属于：

- 节点增强
- 插件模块
- 社区节点能力
- 子后端能力

### 新增一层：协作门

实现前必须明确：

- 谁拥有该模块
- 可否单独 PR
- 主接口在哪里冻结
- 对 Wiki 的文档输出是什么

## 8. 本部分的关键结论

- 现有 SOP 不应废弃，应升级。
- 节点未来应朝“单目录主理解 + manifest-first + 少量标准注册点”演进。
- 插件必须重新定义为“仓内独立模块”，而不是 UI mock 或节点别名。
- 社区节点应继续保留，但不要混同为插件体系本身。
