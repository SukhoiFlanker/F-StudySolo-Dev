# 目标 Monorepo 形态设计

## 1. 目标不是“把所有东西都做成一个技术栈”

本项目更适合的方向不是强行统一成纯 JS workspace，也不是立即拆成大量独立部署服务，而是：

> 保持 Polyglot Monorepo，补齐根级治理层，把不同语言和不同子项目纳入同一套目录边界、任务边界和协作边界。

原因：

- 前端和 Wiki 适合 Node/Next.js。
- 主后端和未来子后端更适合 Python/FastAPI 或其他服务技术。
- `shared/` 已是外部 submodule，不适合假装成普通内部 package。
- 现在最急迫的是协作可拆分，不是技术栈大一统。

## 2. 目标目录形态

建议未来逐步收敛为以下逻辑结构，而不是一次性硬迁移：

```text
StudySolo/
├─ apps/
│  ├─ web/                 # 现 frontend
│  ├─ api/                 # 现 backend
│  ├─ introduce/           # 现 introduce
│  └─ wiki/                # 现 wiki
├─ modules/
│  ├─ workflow-core/       # 工作流领域共享前后端契约
│  ├─ community-nodes/     # 社区节点模块
│  ├─ plugins/             # 仓内独立插件模块
│  └─ agent-gateway/       # 子后端/子 Agent 接口层
├─ shared/                 # 继续保留为 git submodule
├─ supabase/
├─ docs/
└─ tooling/                # 根级任务脚本、质量门禁、CI 支撑
```

这里的关键不是立刻重命名目录，而是先建立“逻辑等价层”。

## 3. 根级治理应补哪些能力

### 3.1 根级任务入口

未来需要有统一入口来做：

- 启动全部开发服务
- 单模块启动
- 全仓测试
- 按影响范围测试
- 文档检查
- 迁移检查

现在的 `scripts/start-studysolo.ps1` 更像本地便捷脚本，不是通用 monorepo orchestration。

### 3.2 根级依赖图

要显式描述以下关系：

- `web` 依赖 `api` 的 HTTP 协议。
- `wiki` 依赖主仓文档和部分共享规范，但不依赖主业务代码。
- `introduce` 只依赖产品与品牌素材，不依赖主执行逻辑。
- `community-nodes`、`plugins`、`agent-gateway` 等模块依赖主平台提供的核心契约。

### 3.3 根级质量门禁

应统一定义：

- 目录命名规范
- 模块公共接口规范
- 不能跨层 import 的边界
- PR 必须补哪些文档
- 哪些变化必须同步更新 API / schema / migration / wiki

## 4. 不建议的方向

### 4.1 不建议立刻把所有插件都做成独立部署服务

原因：

- 运维成本暴涨。
- 鉴权和计费复杂度立刻抬高。
- 当前团队规模与代码状态不支持。

### 4.2 不建议继续保持“目录上分开、治理上完全分散”

原因：

- 这会让 monorepo 只剩“同仓存放”的表象。
- 协作收益无法兑现。

## 5. 未来的模块边界原则

一个模块只有在同时满足以下条件时，才算真正可协作：

- 有清晰的目录边界。
- 有明确的 owner。
- 有稳定的输入输出契约。
- 有独立测试入口。
- 有清晰的文档入口。
- 改动该模块时，不需要顺手改 6 个不相关目录。

## 6. 推荐的分阶段推进方式

### 第一阶段

- 不动大目录名。
- 先建立根级治理文档与质量门禁。
- 先把模块按逻辑分区梳理清楚。

### 第二阶段

- 把最复杂、最容易多人协作冲突的能力抽成独立模块。
- 优先候选：
  - 节点系统
  - 社区节点
  - 子后端接入层
  - Wiki 文档管线

### 第三阶段

- 再考虑目录重命名、根级 task runner、统一 CI pipeline。
