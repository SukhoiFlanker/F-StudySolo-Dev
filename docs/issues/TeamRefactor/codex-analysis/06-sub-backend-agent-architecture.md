# 多 Agent 子后端与 OpenAI 兼容子后端架构

## 1. 目标定义

用户提出的核心目标可以重述为：

> 主平台保留统一入口，同时允许团队成员把特定 Agent 功能封装为独立子后端；主平台只要拿到 API Key，并通过 OpenAI 兼容格式调用，就能将其接入工作流、插件或其他能力模块。

这意味着未来架构需要同时支持两种东西：

- 主后端中的原生节点/服务
- 外接的子后端能力

## 2. 为什么这条路线可行

原因有三点：

1. 当前项目本身就高度依赖 AI provider 抽象。
2. 用户希望多人协作，而子后端是天然的边界单元。
3. OpenAI 兼容接口已经成为事实上的通用接入面。

## 3. 但“OpenAI 兼容”不能只理解成字段长得像

未来若采用这条路线，子后端至少要满足四层兼容：

### 3.1 请求兼容

- `model`
- `messages` / `input`
- 温度、流式、工具等基础参数

### 3.2 响应兼容

- 非流式响应结构
- 流式 chunk 结构
- 错误格式

### 3.3 运行时兼容

- 超时
- 重试策略
- 幂等要求
- 速率限制

### 3.4 平台治理兼容

- API Key 传递方式
- 审计
- 用量统计
- 版本标识

如果只做第 3.1 层，系统会在联调阶段迅速碎裂。

## 4. 推荐的未来拓扑

```text
frontend
  -> main backend
      -> built-in nodes/services
      -> plugin modules
      -> agent gateway
          -> sub-backend A
          -> sub-backend B
          -> sub-backend C
```

其中主后端需要新增一个稳定层：

> Agent Gateway / Capability Gateway

它负责：

- 发现子后端能力
- 统一认证
- 统一请求协议转换
- 统一流式事件映射
- 统一错误归一化
- 统一计量与审计

## 5. 子后端应如何分类

### 5.1 Agent 型子后端

适合：

- 特定领域分析器
- 专用工具编排器
- 复杂提示词/工具链封装器

### 5.2 Plugin 型子后端

适合：

- 第三方平台集成
- 文件处理流水线
- 特殊数据源能力

## 6. 子后端最小契约建议

未来每个子后端至少要公开：

- `id`
- `name`
- `version`
- `capabilities`
- `openai_compatible_base_url`
- `auth_mode`
- `streaming_support`
- `health_endpoint`
- `cost_tag` 或计量说明

主平台必须知道：

- 它能做什么
- 由谁维护
- 如何鉴权
- 是否支持流式
- 如何降级

## 7. 与前端的关系

前端不应直接感知每个子后端的内部实现，而应只感知三类能力：

- 可选择的模型/能力
- 可放置的节点/插件入口
- 可展示的执行事件

这意味着：

- 子后端不能绕开主后端直接向前端输出私有协议。
- SSE 或其他流式事件最终应回到主平台统一事件模型。

## 8. 团队协作上的意义

一旦子后端契约稳定，团队成员就可以：

- 独立开发某个 Agent 子后端
- 独立做该模块测试
- 只通过标准 gateway 接入主系统
- 不必理解整个主后端全部细节

这正是本项目未来多人协作最有价值的边界之一。

## 9. 不建议现在就做的事

- 不建议立刻把现有所有节点外包成子后端。
- 不建议每个子后端都有自己一套前端协议。
- 不建议让子后端直接侵入主前端状态管理。

## 10. 本部分结论

- 子后端路线值得做，但必须先定义统一 gateway 契约。
- “OpenAI 兼容”应被定义为平台级兼容，而不是仅请求字段兼容。
- 子后端应成为团队协作的独立交付单元，而不是临时拼接服务。
