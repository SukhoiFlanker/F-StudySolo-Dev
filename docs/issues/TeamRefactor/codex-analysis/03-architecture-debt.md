# 架构债务与问题清单

## 1. 根级 monorepo 治理债务

### 1.1 现状

仓库在概念上是 monorepo，但在工程治理上仍是多个子项目并排放置：

- 根目录没有统一依赖入口。
- 根目录没有统一任务入口。
- 根目录没有统一构建、测试、lint、发布编排。
- 子项目之间的依赖关系缺少显式图谱。

### 1.2 风险

- 多人协作时，很难快速判断改动影响范围。
- 子项目容易各自形成不同规范。
- CI/CD 和本地开发脚本会逐渐分裂。
- “看起来在一个仓里”不等于“真的可协作”。

## 2. 节点系统存在多源定义债务

### 2.1 当前重复定义点

节点相关事实目前至少分散在以下位置：

- 后端：`backend/app/nodes/_base.py`
- 后端：`backend/app/nodes/__init__.py`
- 后端：`backend/app/api/nodes.py`
- 前端：`frontend/src/types/workflow.ts`
- 前端：`frontend/src/features/workflow/constants/workflow-meta.ts`
- 前端：`frontend/src/components/layout/sidebar/NodeStoreDefaultView.tsx`
- 前端：`frontend/src/features/workflow/components/nodes/index.ts`

### 2.2 重复定义的具体类型

- 节点类型字符串
- 节点分类
- 节点展示名称与描述
- 节点是否需要模型
- 节点输入输出描述
- 节点渲染器映射
- 节点商店分组

### 2.3 结论

后端已经有 manifest 输出能力，但前端没有彻底转向“manifest-first”。

这意味着：

- 现在的节点系统不是单一事实源。
- 以后每新增一个节点，极易同时改 4 到 7 个位置。
- 团队协作时最容易出现“后端能跑、前端不显示”或“前端可放置、后端不支持”的漂移。

## 3. 插件概念混乱债务

### 3.1 当前至少存在四种“插件/扩展”语义

- SOP 中的 C 型插件：独立前后端功能模块。
- 侧边栏中的 `ExtensionsPanel`：当前以 mock 为主。
- `PluginsPanel`：同样是演示式 UI，不是真实安装系统。
- `community-nodes`：实际上更接近“可发布的社区节点扩展能力”。

### 3.2 直接问题

- “插件”这个词现在不能精确表达系统边界。
- 团队成员看到“插件”后，不知道是做节点、做独立模块、做商店 UI，还是做第三方集成。
- 若不先统一术语，后续重构会在概念层面先发生冲突。

## 4. 子后端缺位债务

### 4.1 当前现状

用户希望未来支持“多 Agent 子后端 + 团队成员独立子后端”，但当前系统仍是单主后端：

- 所有核心执行与 API 都在 `backend/`。
- 没有定义主后端和子后端之间的协议。
- 没有定义认证、密钥、流式协议、错误模型、版本兼容约束。

### 4.2 结果

目前所谓“子后端”还只是协作愿景，不是工程结构。

如果现在直接让队友各写一个后端，很容易出现：

- 协议不兼容
- OpenAI 兼容只停留在表面字段兼容
- 流式事件无法统一接入工作流执行界面
- 监控、配额、审计全部断裂

## 5. 文档与真实代码存在漂移

### 5.1 典型案例

- `backend/README.md` 中的 API 前缀与当前真实 `router.py` 已不完全一致。
- `backend/README.md` 中对引擎文件名的描述也与真实文件结构存在出入。
- `wiki/` 在 README 中已被纳入仓库结构，但实际上还未初始化。

### 5.2 结论

文档并非失控，但已经出现“架构复杂度超过文档维护速度”的信号。

这不是写文档不认真，而是说明：

> 当前项目已经需要更强的“事实源优先”和“自动化对齐机制”。

## 6. 团队协作边界不够明确

虽然仓库已经有：

- `CODEOWNERS`
- `pull_request_template.md`
- Issue 模板

但当前仍是单一 owner 主导：

- `CODEOWNERS` 实际上全部指向同一人。
- 还没有按模块拆 ownership。
- 还没有“接口冻结后允许并行开发”的文档机制。
- 还没有把 Wiki、插件、子后端定义为可独立交付单元。

## 7. 当前最需要优先解决的债务排序

### P0

- 节点系统多源定义
- 插件概念混乱
- 子后端缺少统一契约

### P1

- 根级 monorepo 治理缺位
- Wiki 与主项目文档边界未建立
- 文档与代码漂移开始出现

### P2

- 协作 ownership 仍为单人兜底
- 介绍页、Wiki、主前端三者尚未纳入统一体验架构
