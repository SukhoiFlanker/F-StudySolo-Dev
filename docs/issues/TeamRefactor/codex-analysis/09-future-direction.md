# 未来方向分析

## 1. 总方向

未来架构不应继续沿着“单体主系统不断加页面、加节点、加接口”的路线前进，而应转向：

> 主平台稳定化 + 扩展体系模块化 + 协作文档产品化

这三个方向必须并行推进，缺一不可。

## 2. 主平台稳定化

主平台未来应聚焦于四件核心能力：

- 工作流运行时
- 用户与权限
- AI 路由与计量
- 扩展接入网关

所有与这些能力无直接关系的东西，都应尽量从“主系统直接堆叠”转向“模块接入”。

## 3. 扩展体系模块化

未来的扩展体系应至少分成三层：

- 内置节点层
- 仓内插件层
- 子后端/子 Agent 层

这样做的价值是：

- 内置能力继续保持主系统稳定。
- 插件能力可以独立开发与合并。
- 子后端可以由不同成员并行推进。

## 4. 文档产品化

当前文档已经很多，但未来需要把文档分成两类产品：

- 面向设计与重构的内部文档产品
- 面向用户与集成者的 Wiki 文档产品

文档一旦产品化，团队协作才会真正顺畅。

## 5. 不应追求的方向

- 不应追求“所有子项目统一一套技术栈”。
- 不应追求“所有插件都独立部署”。
- 不应追求“任何新能力都先进主系统，后面再拆”。
- 不应继续容忍节点事实在前后端多处重复定义。

## 6. 未来三条最重要的设计原则

### 原则一：单一事实源

同一个事实只能有一个权威定义地。

### 原则二：可独立交付

每个未来模块都应支持独立开发、独立测试、独立 PR。

### 原则三：主平台只保留平台职责

主平台越像“总线 + 契约 + 运行时”，团队协作越容易扩张。

## 7. 本部分结论

未来方向不是“把项目越做越大”，而是“把项目越做越可拆、越可接、越可协作”。
