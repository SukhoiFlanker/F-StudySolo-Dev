# Wiki 与主项目接口分析

## 1. 当前判断

`wiki/` 目前尚未初始化，因此现在讨论的重点不是“怎么改 Wiki 代码”，而是：

> 当 Wiki 未来作为独立子项目存在时，它和主项目之间应该以什么方式连接，才能既利于团队协作，又不制造文档双写灾难。

## 2. Wiki 不应承载什么

Wiki 不应成为：

- 主系统源码注释的替代品
- 主系统架构真相的唯一来源
- 每次发布都手动复制一份的静态说明书

因为这三件事都会迅速失控。

## 3. Wiki 应承载什么

Wiki 更适合承载：

- 用户指南
- 面向开发者的稳定接口文档
- 经过确认的架构说明
- FAQ、接入说明、插件开发指南
- 子后端接入指南

## 4. 主项目文档与 Wiki 的边界

建议采用“双层文档”：

### 4.1 仓库内文档

作用：

- 设计分析
- 未冻结方案
- 重构计划
- 内部决策记录
- 迁移草案

### 4.2 Wiki 文档

作用：

- 已冻结、可对外或可对团队广泛传播的说明
- 面向使用者和集成者的知识

## 5. 推荐的同步模型

不建议双向同步。

建议采用：

> 主仓文档为设计源，Wiki 为发布源。

具体规则：

- `docs/Plans/TNRCodex/` 保留分析与设计过程。
- 当某个接口或模块稳定后，再抽取到 Wiki。
- Wiki 不回写设计决策，只回写面向使用者的稳定说明。

## 6. Wiki 与主项目的接口面

未来建议明确以下接口：

- API 文档发布来源
- 节点/插件/子后端开发指南发布来源
- 版本变更日志来源
- FAQ 和运维知识来源

其中：

- API 文档最好从主项目 schema/route 派生。
- 节点和插件指南应来自升级后的 SOP 与模块文档。
- 子后端接入文档应直接基于 gateway 契约。

## 7. 团队协作上的意义

如果 Wiki 边界明确，那么它非常适合交给单独队友：

- 主项目团队继续迭代核心系统。
- Wiki 团队只围绕稳定接口与用户指南工作。
- 双方通过“接口冻结”和“文档发布清单”交接。

这正好符合当前团队规模下的可行拆法。

## 8. 当前最适合先沉淀到 Wiki 的候选主题

- 用户使用指南
- 工作流基础概念
- 节点分类说明
- API 与 SSE 基础说明
- 插件开发入门
- 子后端接入入门

## 9. 本部分结论

- Wiki 应当成为稳定知识发布层，而不是设计草案仓。
- 主项目文档与 Wiki 应采用单向发布关系。
- Wiki 非常适合作为独立协作子项目交给单独队友负责。
