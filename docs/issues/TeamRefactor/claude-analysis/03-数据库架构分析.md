```

### 4.2 表设计观察

**ss_workflow_runs 表（根据代码推断）：**
- 存储每次工作流执行的状态
- 包含 trace 信息（执行链路）
- 与 `ss_workflows` 通过 `workflow_id` 关联

**ai_usage_ledger 表：**
- 按分钟聚合的使用量（`fn_ss_ai_usage_minute_increment` RPC）
- 支持多维度统计（provider, model, sku, family, vendor, billing_channel）

### 4.3 外键关系（推断）

```

  ├── ss_workflow_runs (1:N)
  ├── ss_workflow_nodes (1:N)
  ├── ss_favorites (1:N, via user_id)

   supabase migration apply

## 6. 数据库与后端/前端类型同步

### 6.1 当前同步方式

**后端：** Pydantic models 在 `backend/app/models/`
```python
class WorkflowRun(BaseModel):
    id: str
    workflow_id: str
    status: str
    # ...
```

**前端：** TypeScript types 在 `frontend/src/types/`
```typescript
interface WorkflowRun {
  id: string;
  workflow_id: string;
  status: string;
  // ...
}
```

**问题：** 两边独立维护，容易出现不一致。

### 6.2 建议：共享类型方案

```
shared/                          # Git Submodule
├── types/
│   ├── workflow.ts             # Workflow 相关类型
│   ├── user.ts                 # User 相关类型
│   └── ai.ts                   # AI/Billing 相关类型
```

```
backend/app/models/             # 从 shared/ 导入
from shared.types import WorkflowRun

frontend/src/types/             # 从 shared/ 导入
import { WorkflowRun } from '@/shared/types';
```

**注意：** shared/ 已存在但主要用于 AGENTS.md。需要在 shared/ 中增加 types/ 目录。
