## P0 - 必须修复的问题

### 1. 跨 Store 耦合：useAIChatStore 直接操作 useConversationStore

**文件：** `frontend/src/stores/use-ai-chat-store.ts:82-96`

```typescript
pushMessage: (role, content) => {
  const entry: ChatEntry = {
    id: crypto.randomUUID(),
    role,
    content,
    timestamp: Date.now(),
  };
  set((state) => ({ history: [...state.history, entry] }));

  // ⚠️ 直接调用另一个 store 的内部状态
  const convStore = useConversationStore.getState();
  if (!convStore.activeId) convStore.createConversation();
  convStore.appendMessage(entry);

  return entry.id;
},
```

**问题：**
- `useAIChatStore` 在 setter 中直接调用 `useConversationStore.getState()`
- 这是隐式的跨 store 依赖，违反 Zustand 最佳实践
- 如果 `useConversationStore` 重构（如改名为 `useSessionStore`），`useAIChatStore` 会悄悄出错

**影响：** 状态同步逻辑分散，难以调试。

**重构建议：**
```
方案A：合并为单一 store（如果两个状态确实需要同步）
方案B：使用事件/发布订阅模式
方案C：在调用方（如 use-stream-chat.ts）处理跨 store 同步，而不是在 store 内部
```

## P1 - 应该优化的问题

### 3. workflow.server.service.ts 与 workflow.service.ts 重复

**文件：**
- `frontend/src/services/workflow.service.ts`
- `frontend/src/services/workflow.server.service.ts`

**重复函数（workflow.server.service.ts 重实现了）：**
- `fetchWorkflowList`
- `fetchWorkflowContent`
- `fetchPublicWorkflow`

**差异：** server 版本使用 cookie-based 认证，client 版本使用 Authorization header。

**问题：**
- 当 `workflow.service.ts` 更新时，server 版本可能不同步
- 两个版本维护成本高

**建议：**
```
方案A：统一为 workflow.service.ts，增加 authMode 参数
      fetchWorkflowList(authMode: 'header' | 'cookie')
方案B：保留两个文件，但提取共享逻辑到 _workflow-base.ts
方案C：确认 server 版本是否真的需要独立，如果不是则删除
```

### 5. CustomEvent Bus 过度使用

**文件：** `frontend/src/features/workflow/components/` (多处)

**使用场景：**
- Canvas tool 切换
- Node placement 触发
- Annotation 模式切换
- 全局 slip 展开/折叠

```typescript
// 示例
window.dispatchEvent(new CustomEvent('canvas-tool-change', { detail: { tool } }));
window.addEventListener('canvas-tool-change', handler);
```

**问题：**
- CustomEvent 是隐式通信，难以追踪数据流
- 无法享受 TypeScript 类型检查
- 事件名称字符串容易拼写错误
- 调试困难（需要在 DevTools 中手动查找事件）

**建议：**
```
方案A：使用 Zustand action 作为显式通信
      useWorkflowStore.getState().setCanvasTool(tool)
方案B：使用 React Context（对于 Canvas 树内的组件）
方案C：保留 CustomEvent 但增加类型定义和事件注册表
```

### 7. services/api-client.ts 与各 service 文件职责重叠

**api-client.ts 提供：**
- `authedFetch` - 自动添加 Authorization header
- `credentialsFetch` - 使用 credentials 模式
- `buildApiUrl` - URL 构建
- `parseApiError` - 错误解析

**但各 service 文件经常直接使用 fetch：**
```typescript
// workflow.service.ts
const res = await fetch(buildApiUrl(`/workflow/${workflowId}/content`), {
  headers: { Authorization: `Bearer ${session?.access_token}` }
});
```

**问题：** 没有统一使用 `api-client.ts`，部分 service 直接用 fetch。

**建议：** 确保所有 API 调用都通过 `api-client.ts`，并增加更多 helper。

## 插件/节点系统现状评估

### 当前 Plugin 系统评估

**优势：**
1. `RENDERER_REGISTRY` 使用注册表模式，易于扩展
2. `community_nodes` feature 已有基本的发布/分享机制
3. Node manifest 从后端 `/nodes/manifest` 动态获取

**问题：**
1. **只写模式**：只能发布节点到社区，无法在本地安装社区节点
2. **无动态代码加载**：节点是 prompt 模板，不是可执行代码
3. **前后端类型不同步**：前端 registry 和后端 registry 独立维护
4. **渲染器限制**：community 节点只能使用预定义的几个 renderer

### Plugin 架构增强方向

```
当前状态：
community_nodes → 发布到数据库 → 其他用户看到

目标状态：
community_nodes → 独立 npm 包或本地模块 → 自动加载到 registry
```

**建议的 Plugin 架构：**
```
frontend/plugins/              # 本地插件目录
├── .gitkeep                   # 团队成员在此添加插件
├── README.md                  # 插件开发指南
└── example/
    ├── plugin.json            # 插件元信息
    ├── renderer.tsx           # React 组件
    └── node-config.ts         # 配置 schema

后端：backend/app/plugins/     # 后端插件
├── __init__.py
├── plugin_registry.py         # 插件发现和加载
└── example/
    ├── plugin.json
    └── node.py
```
