## 2. 团队协作问题分析

### 2.1 当前问题

1. **所有节点在一个目录** - 多人同时开发不同节点会冲突
2. **前后端节点定义不同步** - 后端 NODE_REGISTRY 和前端 RENDERER_REGISTRY 独立维护
3. **节点添加需要改多处** - 后端 node.py、前端 registry、前端 renderer
4. **没有版本管理** - 节点更新没有版本控制

### 2.2 团队协作场景

**场景 A：** 团队成员 A 开发 quiz_gen 节点，团队成员 B 开发 mind_map 节点

**问题：** 两个节点都在 generation/ 目录，可能同时修改 __init__.py 或其他共享文件。

**场景 B：** 团队成员 A 修改了 node.py 的基类，影响所有节点

**问题：** 没有测试隔离，一个节点的改动可能破坏其他节点。

## 3. 重构方案

### 3.1 目录结构重构

**目标：支持多人并行开发，最小化冲突**

```
backend/app/nodes/
├── _base.py                 # 基类（稳定，较少修改）
├── _categories.py
├── _mixins.py
├── _registry.py             # 动态发现（替代 __init__.py 的静态导入）
│
├── nodes/                   # 官方节点（核心功能）
│   ├── trigger_input/
│   ├── knowledge_base/
│   ├── ai_analyzer/
│   ├── quiz_gen/
│   └── ...
│
└── plugins/                 # 社区/团队插件（独立开发）
    ├── _plugins.yaml        # 插件清单
    ├── anki_exporter/
    ├── notion_sync/
    └── ...
```

**关键改进：**
- `plugins/` 目录用于独立开发的插件
- 每个插件是独立子目录，不与其他插件冲突
- `_registry.py` 使用动态发现，不依赖静态导入

### 3.2 节点动态发现机制

**当前：** 静态导入
```python
# nodes/__init__.py
from nodes.input.trigger_input import TriggerInputNode
from nodes.generation.quiz_gen import QuizGenNode
# ... 每次新增节点需要添加一行
```

**目标：** 动态发现
```python
# nodes/_registry.py
import pkgutil
from pathlib import Path

def discover_nodes():
    """动态发现所有节点类"""
    nodes = {}
    for loader, name, ispkg in pkgutil.iterModules(['nodes/nodes']):
        if name.startswith('_'):
            continue
        module = importlib.import_module(f'nodes.nodes.{name}')
        for attr_name in dir(module):
            attr = getattr(module, attr_name)
            if isinstance(attr, type) and issubclass(attr, BaseNode) and attr is not BaseNode:
                nodes[attr.node_type] = attr
    return nodes

# 也扫描 plugins/ 目录
def discover_plugins():
    # ...
```

### 3.3 插件定义规范

每个插件必须包含：

```
backend/app/nodes/plugins/my_plugin/
├── __init__.py              # 导出 node 类
├── node.py                  # 主节点类
├── prompt.md                 # 提示词（如需要）
├── service.py               # 服务层（如需要）
└── plugin.json              # 插件元信息
```

**plugin.json 示例：**
```json
{
  "id": "anki-exporter",
  "name": "Anki Exporter",
  "version": "1.0.0",
  "author": "Team Member A",
  "node_type": "anki_export",
  "category": "output",
  "description": "Export flashcards to Anki format",
  "dependencies": [],
  "frontend": {
    "renderer": "AnkiExportRenderer"
  }
}
```

### 3.4 前端节点注册表重构

**目标：从后端 manifest 自动生成**

```typescript
// services/node-manifest.service.ts（已有）
// 修改为自动生成 RENDERER_REGISTRY

import { RENDERER_REGISTRY } from '@/features/workflow/components/nodes/renderers';

// 从后端 /nodes/manifest 获取节点定义
const manifest = await fetch('/api/nodes/manifest').then(r => r.json());

// 动态注册渲染器
manifest.forEach((node: NodeManifestItem) => {
  if (node.renderer) {
    RENDERER_REGISTRY[node.type] = getRendererComponent(node.renderer);
  }
});

function getRendererComponent(name: string) {
  // 从预定义的映射中获取，或动态导入
  const RENDERER_MAP = {
    'QuizRenderer': QuizRenderer,
    'AnkiExportRenderer': () => import('./anki/AnkiExportRenderer'),
  };
  return RENDERER_MAP[name];
}
```

## 4. 团队协作流程

### 4.1 节点开发流程

```
1. 从主分支创建 feature/节点名
   git checkout -b feat/new-quiz-gen

2. 在 nodes/nodes/ 或 nodes/plugins/ 下开发
   nodes/plugins/my_feature/
   ├── node.py
   └── plugin.json

3. 更新节点 manifest（如需要）
   # 后端自动发现，不需要手动注册

4. 开发前端渲染器
   features/workflow/components/nodes/renderers/
   └── MyFeatureRenderer.tsx

5. PR 合并
   git PR → code review → merge
```

### 4.2 冲突预防

**规则：**
1. 插件目录命名使用 `功能名_作者` 格式
   - `anki_exporter_zhangsan`
   - `notion_sync_lisi`

2. 核心节点（nodes/nodes/）的基类改动需要审批

3. 每个节点有独立的测试文件
   - `nodes/nodes/quiz_gen/test_node.py`

### 4.3 节点版本管理


### 5.3 插件安装机制

```python
# nodes/plugins/manager.py

class PluginManager:
    def __init__(self, plugin_dir: Path):
        self.plugin_dir = plugin_dir

    def load_plugin(self, plugin_name: str):
        """加载单个插件"""
        plugin_path = self.plugin_dir / plugin_name
        spec = importlib.util.spec_from_file_location(
            f"plugins.{plugin_name}",
            plugin_path / "__init__.py"
        )
        module = importlib.util.module_from_spec(spec)
        spec.loader.exec_module(module)
        return module

    def discover_all(self):
        """发现并加载所有插件"""
        for item in self.plugin_dir.iterdir():
            if item.is_dir() and not item.name.startswith('_'):
                self.load_plugin(item.name)

    def get_plugin_manifest(self, plugin_name: str) -> dict:
        """获取插件元信息"""
        with open(self.plugin_dir / plugin_name / "plugin.json") as f:
            return json.load(f)
```

## 6. SOP 更新建议

### 6.1 新增节点 SOP

**Step 0：分类（保持不变）**
- A/B/C 型判断

**Step 1：创建插件目录（新）**
```bash
# 对于 A/B 型节点
cd backend/app/nodes/nodes
mkdir my_new_node

# 对于 C 型或社区插件
cd backend/app/nodes/plugins
mkdir my_new_plugin
```

**Step 2：开发节点**
- 编写 node.py、prompt.md（如需要）

**Step 3：前端渲染器（新）**
- 在 `frontend/src/features/workflow/components/nodes/renderers/` 创建 `MyNodeRenderer.tsx`

**Step 4：更新 manifest（可选）**
- 如有自定义配置，添加 `nodes/nodes/_manifest.yaml` 或 `nodes/plugins/_plugins.yaml`

**Step 5：测试**
- 单元测试 + 手动测试

### 6.2 插件发布 SOP

**Step 1：创建 plugin.json**
```json
{
  "id": "unique-plugin-id",
  "name": "Human Readable Name",
  "version": "1.0.0",
  "author": "Your Name",
  "node_type": "my_custom_node"
}
```

**Step 2：PR 到主项目**
- 创建 PR
- 包含 plugin.json
- 包含测试

**Step 3：Code Review**
- 检查 node_type 冲突
- 检查依赖安全
- 检查权限

## 8. 风险评估

| 阶段 | 风险 | 缓解措施 |
|------|------|---------|
| 动态发现 | 性能开销 | 缓存发现结果 |
| 插件隔离 | 依赖冲突 | 插件使用固定版本依赖 |
| 前后端同步 | 节点类型不一致 | manifest 作为唯一真相源 |
| 向后兼容 | 旧节点迁移成本 | 提供自动迁移脚本 |
