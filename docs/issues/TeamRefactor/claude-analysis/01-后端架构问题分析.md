## P0 - 必须修复的问题

### 1. Bug: loop_runner.py 第 64 行双重赋值

**文件：** `backend/app/engine/loop_runner.py:64`

```python
group_id = group_id = group_node["id"]
```

**问题：** 变量 `group_id` 被赋值两次，第二个赋值覆盖第一个。这是 Python 语法错误但不会抛异常，导致 `group_id` 的值是 `group_node["id"]`，第一个赋值毫无意义。

**影响：** 代码可读性差，可能导致后续逻辑错误。

**修复建议：** 删除第一个赋值，改为 `group_id = group_node["id"]`

## P1 - 应该优化的问题

### 4. API 路由碎片化：Workflow 域分散在 5 个路由文件

**文件：** `backend/app/api/router.py:36-40`

```python
router.include_router(workflow_router, prefix="/workflow", tags=["workflow"])
router.include_router(workflow_social_router, prefix="/workflow", tags=["workflow-social"])
router.include_router(workflow_collab_router, prefix="/workflow", tags=["workflow-collaboration"])
router.include_router(workflow_execute_router, prefix="/workflow", tags=["workflow-execute"])
```

**问题：**
- 同一个 `/workflow` 前缀下有 4 个不同的路由文件
- 开发者需要同时打开 4 个文件才能理解 workflow 域
- 路由注册顺序可能影响中间件应用

**建议方案：**
```
方案A：合并为一个 workflow_router.py（内部用 APIVersi`on 或前缀区分）
方案B：保持分散但增加 workflow/ 子目录，使用 __init__.py 聚合
方案C：按功能复杂度拆分，CRUD/Execution 为核心文件，Social/Collab 可分离
```

```python
router.include_router(ai_router, prefix="/ai", tags=["ai"])                    # /generate-workflow
```

```
ai_catalog.py      → /models/catalog（独立）

```

### 7. ai_router.py 服务边界模糊

**文件：** `backend/app/services/ai_router.py`

**问题：**
- `call_llm_structured` 既做路由选择，又做流式处理
- `call_llm_direct_structured` 也包含完整的流式生成器逻辑
- 实际 HTTP 调用在 `llm_caller.py` 的 `stream_tokens` 和 `call_non_stream`

**当前职责分配：**
| 文件 | 职责 |
|------|------|
| `ai_router.py` | 路由选择（从 config.yaml task_routes）、fallback 链、流式生成器包装 |
| `llm_caller.py` | 实际 HTTP 调用、token 处理 |
| `llm_provider.py` | provider client 管理、pricing |

**问题：** ai_router.py 包含 ~197 行代码，职责过多。流式逻辑分散在两处。

**建议：**
```
ai_router.py     → 只保留路由选择逻辑（resolve_task_route_skus, _build_route_candidates）
llm_caller.py    → 包含流式/非流式调用逻辑
流式生成器        → 独立为 generators/ 目录下的生成器类
```

### 8. usage_ledger 交叉关注点重复

**文件：**
- `backend/app/api/ai.py:46-64`
- `backend/app/api/ai_chat.py:101-233`
- `backend/app/api/ai_chat_stream.py:95-297`
- `backend/app/api/workflow_execute.py`（需确认）

**重复模式：**
```python
usage_request = await create_usage_request(...)
with bind_usage_request(usage_request):
    try:
        # ... 业务逻辑 ...
        request_status = "completed"
    except Exception:
        request_status = "failed"
        raise
    finally:
        await finalize_usage_request(usage_request.request_id, request_status)
```

**问题：** 在 4 个文件中重复，当 usage_ledger API 变化时需要同步修改多处。

**建议方案：**
```
方案A：装饰器
    @track_usage(source_type="assistant", source_subtype="chat")
    async def ai_chat(...)

方案B：依赖注入
    async def ai_chat(..., usage_tracker: UsageTracker)

方案C：中间件（适用于请求级别的 tracking）
```

### 10. config.py 配置扁平化

**文件：** `backend/app/core/config.py`

**问题：**
- 所有配置项平铺在 Settings 类中
- AI 平台的 API keys、Supabase 配置、SMTP 配置、CORS 配置全部同级
- 没有分组，配置多了之后难以维护

**建议：**
```python
class Settings(BaseSettings):
    # Database
    supabase_url: str
    supabase_anon_key: str
    supabase_service_role_key: str

    # AI Providers
    class AIProviders:
        dashscope_api_key: str
        deepseek_api_key: str
        moonshot_api_key: str
        # ...
    ai_providers: AIProviders

    # Email
    class EmailConfig:
        smtp_host: str
        smtp_port: int
        # ...
    email: EmailConfig

    # Security
    class SecurityConfig:
        jwt_secret: str
        cors_origins: list[str]
    security: SecurityConfig
```
